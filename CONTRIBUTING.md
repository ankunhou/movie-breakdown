# 贡献指南

感谢你为电影剧本结构化拆解 CLI 提交改进。提交贡献即表示你有权提供相关代码与材料，并同意贡献按照仓库的 [MIT License](LICENSE) 发布。

参与前请同时阅读 [行为准则](CODE_OF_CONDUCT.md) 和 [安全政策](SECURITY.md)。安全漏洞必须私密报告，不要先创建包含细节的公开 Issue。

## 内容与数据边界

本项目处理的剧本和制作资料可能受版权、保密协议及个人信息保护要求约束。贡献内容必须遵守以下规则：

- 不得提交真实、保密、未公开或未经授权的剧本、对白、分析产物和制作资料；
- 不得提交 API Key、令牌、私钥、`.env` 文件或含敏感正文的日志；
- 测试夹具、文档示例和复现材料只能使用合成、公版或已获得明确授权的内容；
- 使用授权材料时，应在 Pull Request 中说明来源、授权范围和必要的署名要求；
- 示例应保持最小化，不应为了复现缺陷复制整部作品或无关段落。

如果不确定材料能否公开，请使用自编的合成样本。

## 开发环境

项目要求 Python 3.12 或更高版本，并使用 uv 管理环境：

```powershell
uv sync
uv run movie-breakdown --version
```

提交前运行与改动范围相称的检查，完整基线为：

```powershell
uv run --no-cache ruff check src tests
uv run --no-cache ruff format --check src tests
uv run --no-cache pytest -q
```

普通测试不得依赖 DeepSeek 凭据或网络。真实在线契约测试必须显式启用，会产生第三方 API 调用与费用；除非正在验证模型适配，否则不需要运行：

```powershell
$env:MOVIE_BREAKDOWN_RUN_LIVE_TESTS = "1"
uv run --no-cache pytest -q -m live
```

运行在线测试时仍不得把输入内容、密钥或生成产物提交到仓库。

## 代码约定

- CLI、文档、错误提示和用户可见产物使用自然简体中文，文件统一为 UTF-8；
- CLI 只负责交互，领域、应用、模型和基础设施逻辑保持分层；
- 单个 Python 文件不得超过 300 行，超出前先拆分职责；
- Python 模块、公开类、公开函数和公开方法必须编写中文 Google 风格 docstring；
- docstring 有参数时写 `Args`，有返回值时写 `Returns`，调用方需要处理异常时写 `Raises`；
- 有状态、生命周期或可替换实现的能力使用类与 `Protocol`，无状态转换和纯算法保留为函数；
- 结构化模型边界使用 Pydantic，并保留内容指纹、恢复和过期检测语义；
- 新能力和缺陷修复应带自动化测试，不得通过放宽既有安全门禁使测试通过。

更完整的仓库约束见 [AGENTS.md](AGENTS.md)。

## 提交 Issue

提交缺陷前请先搜索已有 Issue。公开报告应包含版本、环境、最小复现步骤、预期行为和实际行为，但必须删去剧本正文、凭据、个人信息、内部路径及敏感日志。

功能建议应说明使用场景、预期输入输出、兼容性影响，以及为什么现有命令或扩展点无法满足需求。较大的架构变化建议先通过 Issue 讨论。

## 提交 Pull Request

1. 从最新主分支创建主题分支；
2. 每次提交聚焦一个可独立审查的能力；
3. 更新相应测试、README、使用手册或变更日志；
4. 在 Pull Request 中说明动机、实现边界、验证命令和结果；
5. 明确列出兼容性变化、模型调用成本变化和数据处理影响；
6. 确认没有加入真实剧本、秘密信息或本地分析产物。

维护者可能要求拆分过大的变更、补充测试或调整接口。合并、发布时间和是否纳入路线图由维护者根据项目范围与维护能力决定。
