# 安全政策

## 支持范围

| 版本 | 安全更新 |
| --- | --- |
| 1.x | 支持 |
| 1.0 之前的版本 | 不支持 |

维护者通常只为最新发布版本提供安全修复。升级、弃用或支持范围发生变化时，本表会同步更新。

## 私密报告漏洞

请优先使用 GitHub 仓库 **Security → Report a vulnerability** 提交私密漏洞报告，即 GitHub Private Vulnerability Reporting。若仓库页面尚未显示该入口，可以发送邮件至 [houankun@hotmail.com](mailto:houankun@hotmail.com)；邮件主题请以 `[SECURITY]` 开头。也可以创建一个不含漏洞细节的普通 Issue，请求维护者启用私密报告，但不要在该 Issue 中披露复现步骤、利用代码或敏感数据。

一份有效报告最好包含：

- 受影响版本、操作系统和 Python 版本；
- 可重复的最小复现步骤；
- 预期行为、实际行为和潜在影响；
- 已知缓解措施或修复建议；
- 是否已经向第三方服务提供商报告。

维护者力争在 7 天内确认收到报告，随后通过私密渠道协调验证、修复和披露时间。响应时间不是服务等级承诺；请在修复发布或双方约定披露前保持报告内容私密。

## 严禁公开提交的内容

无论通过 Issue、Pull Request、讨论区、测试夹具还是日志，都不得公开提交：

- 真实、保密、受版权保护但未经授权的电影剧本或其大段摘录；
- DeepSeek API Key、访问令牌、私钥、`.env` 内容或其他凭据；
- 含个人信息、内部路径、客户数据、模型请求正文或剧本内容的敏感日志；
- 未经授权的分析产物、专家评审记录或制作资料。

若凭据已经泄露，请先在对应平台撤销并轮换，再通过私密漏洞报告告知维护者。测试和复现材料只能使用合成、公版或已获得明确授权的内容，并应尽量缩减到必要范围。

## 报告边界

适合向本项目报告的问题包括路径穿越、任意文件读写、凭据意外持久化、敏感内容进入日志或导出、依赖链风险，以及可绕过本地安全门禁的缺陷。

以下情况通常不属于本项目安全漏洞：

- 单纯的叙事判断偏差、模型幻觉或拆解质量争议；
- DeepSeek 或其他第三方服务自身的中断、计费、隐私政策或账户问题；
- 已明确记录的功能限制；
- 缺少现实世界专业资质、拍摄许可或安全批准。

第三方服务自身的安全问题应同时依照其官方安全政策报告。本项目的 AI 结果不能替代编剧、制片、法务、安全或其他专业人员的判断与批准。
