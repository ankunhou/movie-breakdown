# 项目偏好

- 中文优先：CLI、文档、错误提示和用户可见产物使用自然简体中文，文件统一为 UTF-8。
- 保持清晰分层和克制的目录深度；CLI 只负责交互，领域、应用、模型与基础设施逻辑分离。
- 单个代码文件不得超过 300 行；超出前先拆分模块，并保留自动化约束测试。
- Python 模块、公开类、公开函数和公开方法必须编写清晰的中文 Google 风格 docstring，并由 Ruff 和自动化约束测试检查。类需说明职责、协作边界和关键状态，公共属性不直观时写 `Attributes`；实例方法、`classmethod`、`staticmethod`、属性、`Protocol` 方法和抽象方法均视为公开接口，有参数时写 `Args`，有返回值时写 `Returns`，调用方需处理异常时写 `Raises`；`__init__` 写 `Args` 但不写无意义的 `Returns: None`。复杂或有隐含约束的私有辅助函数也应写 docstring，简单局部纯函数可不重复说明。
- 根据职责选择组织方式：有状态、有生命周期或存在可替换实现的能力使用类与 `Protocol`，优先采用 Strategy、Repository、Factory 等贴合问题的模式；无状态转换和纯算法保留为函数，禁止上帝类和为使用模式而制造空壳抽象。
- 采用 uv、Python、Ruff、Pydantic、Typer、Agno 和 DeepSeek 的主流最佳实践。
- 模型阶段使用结构化类型边界、内容指纹缓存和可恢复产物，不静默复用过期结果。
- 涉及 `screenplay` 目录中真实剧本的回归或验收时，每轮默认只选择一部具有代表性的剧本；除非用户明确要求扩大范围，不得调用模型重跑或改写其他剧本的产物。常规自动化单元测试不受此限制。
- 公开仓库中的测试、示例和文档只能使用完全合成、公版或已取得明确公开授权的材料；不得提交 `screenplay/` 中的真实剧本、模型分析产物、API Key、`.env` 或含敏感原文的日志。
- 边开发边运行与改动范围相称的验证；每个独立能力验证通过后分批提交，避免堆积大提交。
