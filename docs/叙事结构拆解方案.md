# 电影剧本叙事结构拆解 CLI 方案

## 1. 文档状态

- 状态：叙事结构拆解 `1.0.0` 稳定基线已封版；已使用单部经授权的代表性剧本完成专家抽检、人工修正、复核与全部稳定版门禁验收
- 当前目标：维护叙事稳定基线，确保后续扩展不破坏现有证据、恢复和封版语义
- 后续目标：在稳定的场景与证据模型之上扩展制片元素拆解
- 技术栈：uv、Python、Ruff、Pydantic、Typer、Agno、DeepSeek V4 Pro

## 2. 产品定位

本项目是一个可恢复、可追溯、可验证的电影剧本拆解 CLI。它把剧本处理为分阶段的结构化产物，而不是通过一次大模型调用直接生成不可复查的长篇报告。

第一阶段聚焦叙事结构，包括：

- 场景切分与场景摘要；
- 人物、地点和关键事件识别；
- 人物关系与人物弧光；
- 主线、支线与情节因果；
- 三幕结构、序列、节拍和关键转折；
- 冲突、悬念、伏笔与回收；
- 主题、母题与叙事节奏；
- 所有重要结论到剧本文本位置的证据引用。

第一阶段不实现完整的制片拆解、预算估算或拍摄计划，但底层场景模型保留必要的扩展能力，避免后续重复解析剧本。

## 3. 核心原则

### 3.1 确定性流水线优先

应用层显式定义处理阶段、依赖关系、重试和恢复规则。Agno 用于封装结构化模型调用，不负责决定整个任务如何推进。

### 3.2 结构化类型边界

阶段输入和输出均使用 Pydantic 模型。模型输出只有通过 Pydantic 校验后才能成为正式产物；合法 JSON 不等于业务结构有效。

### 3.3 证据优先

重要结论必须尽可能关联剧本中的场景、页码、行号或文本范围。模型无法从剧本确认的信息应明确标记为未知或推测，不允许补写成事实。

### 3.4 内容指纹缓存

缓存不能只依赖文件名。源文本、上游产物、Prompt、Schema、阶段版本、模型和关键参数任一改变时，都应使相关缓存失效。

### 3.5 可恢复执行

每个阶段独立保存状态和产物。任务中断或单个场景失败后，`resume` 只重跑未完成、失败或过期的部分。

### 3.6 CLI 保持轻薄

Typer 命令只负责参数解析、用户交互、进度显示和退出码，不承载领域规则、模型调用细节或文件存储逻辑。

## 4. 叙事稳定基线范围

### 4.1 包含

- UTF-8 TXT、Markdown；
- 可直接提取文字的 PDF；
- 中文剧本优先，兼容常见英文场景标题；
- 场景切分与人工可检查的规范化文本；
- 逐场叙事分析；
- 人物、地点、事件归一；
- 人物关系和人物弧光；
- 覆盖全部已归一人物的确定性分级档案；
- 区分剧本呈现、角色转述、分析推断与未知的人物小传；
- 三幕结构与关键节拍分析；
- 主线、支线、伏笔与回收分析；
- JSON、JSONL 和 Markdown 导出；
- 内容指纹缓存、失败重试和断点恢复；
- 无模型参与的一致性校验；
- 带分析指纹的专家抽检、人工修正和同目标复核；
- 不调用模型的叙事稳定版发布门禁。

### 4.2 暂不包含

- 扫描 PDF 的 OCR；
- Final Draft FDX 和 DOCX；
- 完整制片元素表、预算和拍摄排期；
- Web UI；
- 向量数据库；
- 多 Agent 自主协作；
- 长期对话记忆。

## 5. 总体处理流程

```mermaid
flowchart LR
    A["导入剧本"] --> B["文本规范化"]
    B --> C["场景切分"]
    C --> D["逐场结构化分析"]
    D --> E["实体归一与关系合并"]
    E --> F["全局叙事结构分析"]
    F --> G["全人物确定性分级档案"]
    G --> H["核心人物可恢复小传分析"]
    H --> I["一致性校验"]
    I --> J["JSON / Markdown 导出"]
    J --> K["语义风险信号与专家抽检"]
    K --> L["指纹绑定的人工修正"]
    L --> M["重新校验与导出"]
    M --> N["新指纹专家复核"]
    N --> O["叙事稳定版门禁"]
```

### 5.1 导入剧本

生成源文件指纹，记录格式、编码、页数和导入时间。源文件进入项目目录后只读使用，后续分析基于规范化产物。

### 5.2 文本规范化

统一换行、空白和常见剧本标记，同时保留原始页码、行号或段落映射。规范化过程不得丢失回指原文所需的信息。

### 5.3 场景切分

优先使用确定性规则识别数字场次、`场景：` 元数据、`INT.`、`EXT.`、`内景`、`外景`等场景标题，并计算场景数量、平均场长和回退状态。只有本地结果不可靠时，`auto` 策略才调用 Agno 与 DeepSeek：PDF 抽取前三页和后三页，TXT/Markdown 抽取首尾各约 150 行，生成严格类型的场景格式画像和候选正则。

模型正则必须经过本地安全与质量校验，包括行首锚点、长度限制、禁止回溯引用和环视、标题示例命中、场景数量和平均长度。验证失败时不得直接执行或静默接受。CLI 同时提供 `auto`、`local` 和 `model` 三种格式识别策略。

场景切分结果必须允许用户直接检查，不能与后续叙事判断耦合。

### 5.4 逐场结构化分析

每个场景独立分析，主要输出：

- 场景概述；
- 出场人物；
- 人物目标与阻力；
- 核心冲突；
- 关键行动和事件；
- 场景开始与结束时的状态变化；
- 新增信息和悬念；
- 伏笔、回收候选；
- 对主线或支线的作用；
- 相关证据与不确定项。

逐场处理可以受控并发，但产物顺序必须与剧本场景顺序一致。

### 5.5 实体归一与关系合并

将别名、简称和称谓归并到统一实体，例如判断“老王”“王警官”“王建国”是否为同一人物。归并结果需要记录：

- 标准名称；
- 所有别名；
- 归并证据；
- 置信度；
- 无法自动判断的冲突。

### 5.6 全局叙事结构分析

全局分析主要使用已经验证的场景摘要、事件和实体关系，而不是每次重新输入完整剧本。第一版默认提供三幕结构分析，并为后续结构理论预留插件式接口：

- `three-act`：三幕结构；
- `sequence`：八序列等序列分析；
- `save-the-cat`：节拍表；
- `none`：不套用指定理论，只做文本内生结构总结。

当前稳定基线完整实现 `three-act`，其他框架随后增加。

### 5.7 全人物分级档案

“全部人物”指全局分析完成别名归一后 `entities.characters` 中的全部人物实体，不把逐场输出中的每一个原始称谓直接当作独立人物。无法可靠归并的原始称谓仍属于实体归一质量问题，后续可增加“待确认人物候选”清单，不能为了表面覆盖率强行建档。

档案阶段不调用 DeepSeek，而是复用已验证的实体、场景、事件、关系和人物弧光，按全局人物顺序为每个人物生成一份 `CharacterDossier`。档案包含标准名称、别名、摘要、首次出场、全部有效场景、参与事件、人物关系、原文证据、重要度信号和中文分级理由。

默认 `rule-based-v1` 策略分为四层：

- `core`（核心人物）：拥有有效场景，且存在人物弧光或出现场景数位列全剧前三；
- `supporting`（重要配角）：出现场景数达到 `max(5, 全剧场景数的 5% 向上取整)`，或参与事件数达到同口径阈值，或至少出现两场且与核心人物存在直接关系；
- `functional`（功能人物）：未进入前两层，但至少有有效场景、事件或人物关系；
- `background`（背景/索引人物）：仅保留归一实体索引，暂时没有可验证的场景、事件或关系引用。

分级规则采用 Strategy 与 `Protocol` 边界，便于以后替换为面向剧种或项目的策略。当前策略是确定性的：相同上游产物必然得到相同层级、顺序和理由，不新增模型 token。

### 5.8 声明级人物小传

人物小传是独立于全人物档案的可恢复模型阶段，只覆盖档案中的 `core` 人物。每个人物拥有单独缓存键和 JSONL 记录；单个人物失败后只重试该人物。旧项目新增档案阶段时会先确定性补齐全部档案，再沿用兼容的逐人物缓存，不强制重新消耗已有小传 token。

人物小传不输出无法区分来源性质的权威长段落，而是由至多十二条声明组成。每条声明必须携带原文证据、置信度和以下依据类型之一：

- `observed`：动作、场面或舞台说明直接呈现；
- `reported`：角色对白、信件、旁白等作出的陈述，必须记录陈述来源，不能自动视为客观事实；
- `inferred`：依据行为、关系或跨场变化作出的分析推断，必须记录可审查的推断依据；
- `unknowns`：输入没有提供的年龄、前史、秘密等方面，不能伪装成声明。

人物上下文包含归一实体、人物弧、相关关系和事件、人物出现的逐场分析，以及最多八个稳定选择的原文场景。原文优先覆盖首次和末次出场、弧光转折、关键关系与实体证据，再按全片位置均匀补足。代表性台词只能逐字来自这些原文场景，无法确认说话人时必须为空。

### 5.9 一致性校验

校验阶段不调用模型，至少检查：

- 场景编号和顺序连续性；
- 引用的场景、人物、事件是否存在；
- 证据范围是否有效；
- 伏笔与回收引用是否形成合法关系；
- 人物关系变化是否引用有效事件；
- 分级档案是否与全部归一人物一一对应、顺序一致且满足当前策略；
- 档案的场景、事件、关系、首次出场、统计信号和归属人物是否一致；
- 完整小传是否精确覆盖 `core` 人物，关系与场景引用是否有效；
- 人物声明和代表性台词的行号与原文摘录是否可核验；
- 是否存在重复 ID、悬空引用或空关键字段；
- 逐场覆盖率是否达到要求。

### 5.10 叙事语义质量评测

一致性校验不能证明主题、人物弧光、事件因果和转折点解释符合创作语境，因此另设不阻断正式导出的 `review` 旁路：

- 自动检查重要结论的证据覆盖和跨场支撑；
- 检查幕归属重复、幕内连续性和幕间顺序；
- 检查事件因果、伏笔设置与回收、人物弧光转折的场景时序；
- 检查人物小传声明的证据、转述归因、推断依据与持续性推断的多场支撑；
- 按首中尾、三幕转折和高风险结论生成稳定抽检样本；
- 为每个目标附加证据和完整场景上下文；
- 使用五分制维度评分和 supported、partially_supported、unsupported、uncertain 结论；
- 使用分析指纹和评分标准版本拒绝过期人工答案。

自动结果必须称为“代理信号”或“风险信号”，不得汇总成虚假的叙事正确率。人工抽检允许部分完成，未评分项目不进入维度均分。

### 5.11 专家修正、复核与封版

`review` 不阻断普通导出，但 `finalize` 会把专家结果纳入稳定版门禁。完整闭环为：

1. 使用稳定抽样生成带分析指纹的专家评审表；
2. 导入完整或部分专家答案，记录评审者身份、评分、结论和修改建议；
3. 将可执行建议整理为结构化修正集，并绑定源指纹、基础分析指纹和专家答案；
4. 先执行 `correct --dry-run`，在零写入条件下检查路径白名单、类型、证据、目标值和修正后全量一致性；
5. 激活修正后保存原始分析、正式修正快照、活动修正集、专家答案与指纹绑定的确定性审计回执，并重建正式导出；
6. 为修正后的新分析指纹重新生成抽检模板，核对新旧 `target_id`，确保高风险修正目标仍被覆盖，再填写并导入复核答案；不允许用旧分析指纹的答案蒙混通过；
7. 使用 `finalize` 执行结构校验、指纹、评审身份、抽检规模、完成度、结论、维度评分和风险接受说明八项门禁。

人工修正必须是确定性的，不调用模型，不改写原始模型产物。项目恢复时会重放仍与基础分析匹配的活动修正；上游语义变化后必须明确失效，不能静默复用。

## 6. 分层架构

```text
src/movie_breakdown/
├── cli.py             # Typer 主入口
├── cli_*.py           # 质量、封版、渲染与 CLI 类型
├── application/       # analyze、resume、validate、export 用例
├── domain/            # Pydantic 领域模型和纯业务规则
├── pipeline/          # 阶段定义、依赖图和执行器
├── infrastructure/
│   ├── llm/           # Agno、DeepSeek 适配与重试
│   ├── parsers.py     # TXT、Markdown、PDF 解析
│   ├── scene_splitter.py
│   ├── storage.py     # 项目产物、缓存和原子写入
│   └── fingerprint.py
└── config.py          # 环境变量和运行配置
```

依赖方向：

- `domain` 不依赖 Typer、Agno 或具体文件系统；
- `application` 编排领域能力并依赖抽象接口；
- `pipeline` 管理阶段状态和依赖关系；
- `infrastructure` 实现模型、解析器和存储接口；
- `cli` 调用应用用例，不直接调用 Agno。

## 7. 核心领域模型

第一版建议包含：

- `SourceDocument`：源文件信息与总指纹；
- `SourceSpan`：页码、行号、段落和文本范围；
- `Evidence`：结论引用的来源位置与简短摘录；
- `Screenplay`：剧本元信息和场景索引；
- `Scene`：场景标题、正文、顺序和来源范围；
- `SceneAnalysis`：逐场叙事分析结果；
- `Character`：人物、别名和人物属性；
- `Location`：地点和别名；
- `StoryEvent`：事件、参与者、因果和顺序；
- `CharacterRelation`：人物关系及其变化；
- `CharacterArc`：人物目标、转折与最终状态；
- `CharacterDossier`：覆盖单个归一人物的分级资料卡、引用和分级理由；
- `CharacterImportanceSignals`：场景、事件、关系、弧光和高频排名信号；
- `CharacterDossierCatalog`：保持全局人物顺序的全人物档案目录；
- `CharacterBiographyClaim`：带依据类型、置信度和证据的单条人物声明；
- `CharacterBiography`：人物概览、声明、未知项、关键关系和代表性台词；
- `BiographyAnalysisRecord`：按人物保存的缓存、失败和 token 用量记录；
- `PlotThread`：主线或支线及覆盖场景；
- `Beat`：叙事节拍和结构作用；
- `ForeshadowingLink`：伏笔与回收关系；
- `StructureAnalysis`：全局结构分析；
- `ValidationIssue`：错误、警告和定位；
- `RunManifest`：执行阶段、指纹、状态和成本信息。

领域模型约束：

- 默认禁止未知字段；
- 实体 ID 与展示名称分离；
- 每类产物包含 `schema_version`；
- 未知、未提及和低置信度必须可以显式表达；
- 枚举只用于相对稳定的分类；
- 重要判断尽可能包含 `evidence`。

## 8. DeepSeek 与 Agno 集成

默认模型使用 `deepseek-v4-pro`。不使用即将停用的 `deepseek-chat` 和 `deepseek-reasoner` 作为默认名称。

每类分析任务使用一个职责单一的 Agno `Agent`，例如：

- 场景分析 Agent；
- 实体归一 Agent；
- 人物弧光 Agent；
- 人物小传 Agent；
- 全局结构 Agent。

这些 Agent 不自主安排任务，也不直接读写项目文件。应用层向它们传入明确输入，并接收指定 Pydantic Schema 的结果。

### 8.1 结构化输出防线

DeepSeek JSON Output 只保证输出为合法 JSON，不保证完全遵守业务 Schema。因此采用以下流程：

1. Prompt 明确要求输出 JSON，并给出目标结构；
2. Agno 设置对应的 `output_schema`；
3. 启用 DeepSeek JSON Output；
4. 使用 Pydantic 严格校验；
5. 校验失败时，把精简后的错误反馈给模型修复；
6. 最多重试 2 至 3 次；
7. 最终失败时保存失败信息，不生成伪成功产物。

### 8.2 思考模式

建议按阶段配置：

- 文本清洗和简单抽取：不开启或降低使用强度；
- 实体归一、人物弧光、因果和全局结构：开启思考模式；
- 具体参数由配置文件控制，并写入产物元数据。

不在正式产物中保存或展示模型的完整推理内容，只保存最终结构化结论、证据、用量和诊断信息。

## 9. 缓存和恢复

缓存键至少由以下内容组成：

```text
输入内容指纹
+ 上游产物指纹
+ 阶段版本
+ Prompt 指纹
+ Pydantic Schema 指纹
+ 模型名称
+ 影响结果的模型参数
```

任何一项变化，都必须使相关缓存失效。禁止仅根据目标文件存在就跳过阶段。

每个产物记录：

- `schema_version`；
- `stage_version`；
- `source_fingerprint`；
- `upstream_fingerprints`；
- `prompt_fingerprint`；
- `schema_fingerprint`；
- 模型和关键参数；
- 创建时间；
- token 用量和估算费用；
- 成功、失败或过期状态；
- 错误类型和可恢复信息。

写文件采用临时文件加原子替换，避免中断后留下看似完整的损坏产物。

逐场与逐人物失败记录保留实际模型尝试次数和 token 用量；同一缓存键恢复后，历史失败成本与新调用成本累计写入最终记录。模型已经返回但证据归一化或 ID 校验失败时，也使用该次已完成调用的真实成本，不记为零。

缓存恢复允许执行不改变分析语义的确定性迁移，例如去除无法在当前剧本定位的证据、把误写的人物 ID 映射为已存在的关系 ID，以及补齐前后场景明确属于同一幕的内部遗漏。只要输入变化、迁移后仍有确定性错误或幕边界存在歧义，就保持缓存失效并重跑对应模型阶段。

## 11. CLI 初步设计

```powershell
movie-breakdown analyze 剧本.pdf --project ./项目目录
movie-breakdown status ./项目目录
movie-breakdown resume ./项目目录
movie-breakdown validate ./项目目录
movie-breakdown export ./项目目录 --format markdown
movie-breakdown review ./项目目录 --sample-size 16
movie-breakdown correct ./项目目录 ./专家修正.json --answers ./专家答案.json --dry-run
movie-breakdown correct ./项目目录 ./专家修正.json --answers ./专家答案.json
movie-breakdown finalize ./项目目录
movie-breakdown doctor
```

首次分析可额外指定 `--format-detection auto|local|model`。当前稳定基线的 `--framework` 只开放 `three-act`；其他框架仍保留在领域扩展边界中，待后续实现各自的 Schema、Prompt 和校验规则后再开放。

命令职责：

- `analyze`：创建项目并执行完整流水线；
- `status`：展示阶段状态、缓存命中和失败原因；
- `resume`：继续未完成、失败或过期阶段；
- `validate`：执行本地一致性校验；
- `export`：从已验证产物生成报告；
- `review`：离线生成语义风险信号和人工抽检表，可合并带指纹的专家答案；
- `correct`：严格预览或激活指纹绑定的结构化人工修正，全程不调用模型；
- `finalize`：执行八项叙事稳定版门禁并导出机器和可读报告；
- `doctor`：检查 API Key、模型连通性、目录权限和依赖状态。

CLI 约定：

- 用户可见文本使用简体中文；
- 提供稳定退出码；
- 支持 `--json` 输出机器可读的状态和错误；
- 模型调用期间只显示阶段进度，不把未校验的流式内容当成结果；
- API Key 仅从环境变量读取，绝不写入产物和日志。

### 里程碑 1：项目骨架

### 里程碑 3：逐场叙事分析

- 场景级 Markdown 预览。

### 里程碑 4：全局叙事结构

- 人物、地点、事件归一；
- 人物关系和弧光；
- 主线、支线、伏笔与回收；
- 三幕结构和关键节拍；
- 完整报告导出；

## 15. 叙事稳定基线验收标准

一个版本满足以下条件后，才视为叙事结构拆解稳定基线完成：

1. 能导入一部标准文本型电影剧本并生成可人工检查的场景列表；
2. 中断后能够恢复，已经验证的场景不会被无故重新调用模型；
3. 修改剧本、Prompt、Schema 或模型配置后，相关旧缓存会被识别为过期；
4. 每个场景都能生成通过 Pydantic 校验的结构化分析，失败场景有明确状态；
5. 人物、事件、关系和结构分析不存在悬空引用；
6. 每个已归一人物都有且只有一份可解释的分级档案；
7. 关键全局结论能够回指相关场景和证据；
8. 可以导出稳定 JSON 和可读 Markdown 报告；
9. 专家答案、人工修正和正式分析均受内容指纹约束，旧结果不会静默复用；
10. 修正后通过新指纹复核及八项稳定版门禁，保留结论均有具体风险接受说明；
11. 本地测试、Ruff 检查和文件行数约束全部通过。

## 16. 已确认的决策

- 先做叙事结构拆解，再做制片元素拆解；
- 使用确定性流水线编排，不采用自由运行的多 Agent 系统；
- 使用 DeepSeek V4 Pro 作为默认模型；
- 使用 Pydantic 作为结构化结果的最终有效性边界；
- 使用内容指纹缓存和阶段产物实现可恢复执行；
- CLI、领域、应用、模型适配和存储逻辑分层；
- 第一版默认实现三幕结构分析。
- 每轮真实模型验收只使用一部明确指定的剧本，不批量消耗其他剧本 token；
- AI 模拟专家必须明确标识为非真人，不得伪造自然人身份或签字。

## 17. 待后续确认
