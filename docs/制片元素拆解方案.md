# 制片元素拆解方案

## 1. 目标与当前状态

制片元素拆解是在叙事稳定基线之后增加的独立能力。它从已经确定性切分的剧本场景中提取可追溯的制作需求，为制片、导演、现场统筹及各专业部门提供首轮拆解底稿。

当前首版已完成代码、伪模型端到端测试，并使用单部经授权的代表性剧本完成 DeepSeek 全流程验收。技术流水线、恢复、结构校验和导出门禁已经通过；AI 模拟制片专家只能验证评测路径，不能替代真人专家与各专业角色的安全批准。因此它属于“技术基线完成、专业门禁需按项目单独完成”的 `0.x` 初稿，不与叙事 `1.0.0` 稳定声明混用。

在这个 `0.x` 基线上，制片阶段增加一套纯本地的“规划—评审—修正—封版”闭环。以下章节定义该闭环的实现契约和通过条件；具体项目是否达到评测封版或专业稳定版，只能由当前指纹绑定的门禁产物说明，不能在实跑前预先声明。

基础拆解只回答：剧本画面或声音要求准备什么、哪些场景可能较复杂、哪些关键信息尚未提供。它不自动回答预算、拍摄天数、采购数量、供应商、演员人选和具体拍摄方案；后续规划层只能记录已经由人工作出的数量与安全决定。

## 2. 独立边界

制片流水线只读复用主项目的两个事实：

- `project.json`：核对主项目身份，并复制模型相关配置；
- `artifacts/scenes.json`：读取场景原文、行号、标题和来源指纹。

基础制片模型流水线明确不读取主项目的叙事产物：

- `scene_analysis.jsonl`；
- 人物、事件、关系和三幕结构；
- 人物分级档案与人物小传；
- 专家答案、人工修正和稳定版门禁。

本地制片闭环只读取 `production/` 作用域内自己的评审答案、修正 generation 和发布门禁，不读取或复用叙事阶段的同名结论。

所有制片配置、运行状态、缓存、产物和导出均位于主项目的 `production/` 子目录。叙事 `manifest.json` 不会增加制片阶段，也不会因制片运行而改变。

## 3. 分层与组织方式

代码继续遵守领域、应用、基础设施、流水线和 CLI 分层：

- 领域层：严格 Pydantic Schema、枚举、数量边界和独立项目配置；
- 应用层：`ProductionAnalyzer`、`ProductionCatalogBuilder` 等 Protocol，以及规划、评审、修正、分级发布、验证和导出服务；
- 基础设施层：Agno/DeepSeek 适配器与 `ProductionStore`；
- 流水线层：逐场恢复、总表、校验和导出状态机；
- CLI：只做参数、输出和退出码，不承载领域逻辑。

有状态、存在生命周期或可替换实现的能力使用类和 Protocol；Unicode 规范化、证据去重、数量峰值等无状态算法保留为纯函数。这样既避免上帝类，也不会为了模式制造空壳抽象。

## 4. 逐场 Schema

每个 `SceneProductionAnalysis` 包含：

1. 场景设置
   - 原始标题；
   - 地点与子地点；
   - 内景、外景、混合或未知；
   - 黎明、日、黄昏、夜、连续或未知；
   - 天气要求。
2. 演员需求
   - 画面出演、仅声音、照片/录像、替身或其他；
   - 可选稳定人物 ID；
   - 表演或专业要求。
3. 群演需求
   - 群体名称和描述；
   - 数量上下界和确定程度；
   - 特殊技能。
4. 制片元素
   - 服装、妆发、手持道具、陈设、车辆、动物；
   - 动作特技、实拍特效、视效、特殊设备；
   - 现场声音/音乐及开放兜底类别；
   - 数量、关联演员、状态/连续性和特殊要求。
5. 场景复杂度
   - 1—5 分及固定等级映射；
   - 群演、地点物流、服化道、动作安全、技术特效、连续性/排期等证据因素。
6. 待确认事项
   - 剧本未说明的信息；
   - 该缺口可能影响的制片判断。

场景设置、演员、群演、元素和复杂度因素均必须带原文证据。推断性需求必须明确标记 `inferred` 并填写推断依据。

## 5. 模型调用与恢复

`AgnoProductionAnalyzer` 使用独立 Prompt 和 `SceneProductionAnalysis` Schema，每个场景单独调用 DeepSeek。Agent 无记忆、无工具，结构化失败由共享 Agno 客户端携带精简错误重试。

缓存键只包含真正影响该场制片结果的内容：

- 场景 ID、标题、全文与来源行号；
- 制片 Prompt 指纹；
- 制片 Schema 指纹和阶段版本；
- 制片契约版本；
- 模型、思考开关和推理强度。

它不包含任何叙事分析产物。单场失败会保留错误、实际尝试次数和 token；`production resume` 只重试失败或缓存已过期的场景。移动来源行号也会使对应场景缓存失效，避免复用旧证据位置。

## 6. 确定性基础总表

总表由本地 `ConservativeProductionCatalogBuilder` 生成，不再调用模型。合并键只做 NFKC、Unicode 空白折叠和大小写折叠，不删除标点，不做同义词、简称或模糊匹配。

- 地点：地点名 + 子地点；
- 演员：优先稳定人物 ID，无 ID 时保守使用名称；
- 群演：群体名称 + 描述；
- 元素：类别 + 名称，`other` 额外包含子类型。

稳定目录 ID 由类别和完整规范键计算，不依赖第一个场景或局部需求 ID。所有来源需求都保存为 `scene_id/local_id`，证据只做完整字段精确去重。

`peak_quantity` 是跨场景最大同时需求，不是跨场求和：

- 同单位时保守合并上下界；
- 任一来源为估算时保留估算标记；
- 存在未知上界时结果最多只能声明最低数量；
- 单位不同则降为“单位待确认”；
- 不换算“人/名/位”“辆/台”等近义单位。

## 7. 本地制片规划

规划层消费已经通过一致性校验的共享场景和逐场制片结果，不重新读取密钥，也不调用 Agno 或 DeepSeek。它把模型产出的“逐场需求候选”转换为可以继续人工确认的拍摄单元、资源身份、数量语义和安全范围；模型不能在这个阶段把推断升级为专业决定。

### 7.1 拍摄单元

`ShootingUnit` 是一个剧本场景内部，可分别确认地点、时段、资源和风险的最小拍摄语义单元。它不是镜头，不是拍摄日，也不是通告单或排期中的摄制组单元。

- 每个场景至少有一个单元，单元按稳定序号连续、无空档地覆盖该场全部原文行；
- 本地构建器只根据明确的地点/时段变化、蒙太奇节拍、动作阶段、插入画面或字幕等受控信号生成保守基线；
- 高复杂度长场景、`continuous` 时段和疑似蒙太奇仍只有一个单元时，系统生成“疑似漏拆”强制评审目标，而不是静默相信基线；
- 人工修正以“整场全部单元”为原子替换作用域，同时重建该场资源出现项、地点资源和风险范围，避免局部改动留下悬空引用；
- 每个单元保留来源行范围、拆分原因和逐字证据，后续资源与安全审批都绑定单元 ID。

### 7.2 跨场实体三层模型

资源类别、连续性实体和场内出现项必须分层，不能把同名字符串直接当作已经确认的同一对象：

| 层级 | 职责 | 典型例子 |
| --- | --- | --- |
| `ProductionResourceClass` | 记录规范名称、别名、标准单位、资源类别及身份范围 | “军马”类、某型号车辆类、某角色类 |
| `ProductionEntity` | 记录跨多个出现项保持同一身份或连续性状态的具体对象 | 同一匹英雄马、同一辆剧情车辆、同一件英雄道具 |
| `ResourceOccurrence` | 记录资源在某场、某拍摄单元中的一次可追溯出现 | 第 12 场第二单元中该马的出现和状态 |

`identity_scope` 明确区分可互换资源 `fungible`、需追踪同一身份的 `continuity` 和群体状态 `cohort`。确定性归一只能生成类别和候选实体；连续性实体在结构化评审确认前保持 `unresolved`，冲突则标记 `conflicted`。`confirmed` 必须保留确认依据、别名、出现项和合并前重定向 ID，并用 `basis` 区分 AI 模拟评审与真人确认，不能通过模糊匹配静默合并。未确认的连续性实体会阻断 `catalog_ready`。

### 7.3 数量事实、人工计划与单位

剧本事实和制作决定使用不同模型，禁止把模型估算或采购经验反写成“剧本写明”：

| 层级 | 可以表达 | 不可以表达 |
| --- | --- | --- |
| `QuantityFact` | 原文支持的上下界、标准单位、数量角色、父子关系、逐字证据 | 无证据的模型估算、采购备份量 |
| `PlannedQuantity` | 人工决定的实拍、数字扩充、备份或采购数量，以及评审者、决定 ID、输入指纹和理由 | 修改原始数量事实或冒充剧本证据 |

数量事实区分 `total`、`subset`、`per_member`、`event` 和 `screen_scale`。状态子集和每成员数量必须引用父数量；互斥子集的已知最小值之和不得超过父项上界，避免把“总人数”和“其中伤员”重复相加。确定性派生值必须列出来源数量 ID。

自由文本单位同时保留 `raw_unit`，只有能够安全映射时才进入 `person`、`animal`、`vehicle`、`item`、`set`、`pair`、`costume`、`weapon`、`device`、`event`、`shot` 或 `location` 等标准代码。系统不做未经业务确认的单位换算；`unknown` 会阻断 `catalog_ready`。模型估算或证据中找不到对应数字时，原表达只保留在审计字段中，事实上下界必须为空。

人工计划数量只能通过指纹绑定的结构化人工修正写入，且单位必须与资源类别一致。系统不会根据剧本数量自动计算实拍、数字扩充、租赁、采购或备份数量。

### 7.4 高危元素强制安全复核

确定性安全规则会从逐场结果和资源出现项重建高危候选，覆盖枪械、烟火、明火、刀具打斗、车辆动作、动物动作、高空吊挂、水下/溺水、群体动作、未成年演员、极端环境、人体伤效等范围。风险候选携带触发规则、拍摄单元、资源、逐字证据、必需控制项、禁止方法、所需专业角色和范围指纹。

- 高危候选不能由模型、普通评审答案或人工修正静默删除；校验会重建候选并比较范围指纹；
- “实弹”或危险活体动作等默认实现必须由 `SafetyMethodDecision` 明确否决，并写明安全替代边界；决定过期或缺失会阻断 `catalog_ready`；
- 每个风险的每个必需专业角色都要对同一范围指纹给出唯一 `SafetyApproval`；范围变化后旧批准自动失效；
- AI 模拟评审和未核实人员只能要求修改、要求专业复核或否决，不能批准、附条件批准或把风险标记为不适用；
- 只有声明为 `qualified_professional` 的评审者才能形成 Schema 层的批准。项目负责人仍须在系统外核验其真实身份、资质和权限，字段声明本身不是资质认证；
- `evaluation_ready` 仍可保留“尚未开拍就绪”的限制；只有全部指定角色的合格批准和其他门禁同时通过，规划才可能达到 `shoot_ready`。

## 8. 两级本地校验与三级准备度

基础拆解校验完全不调用模型，阻断以下问题：

- 场景覆盖率不是 100%；
- 逐场记录失败、重复或引用未知场景；
- 场景标题、需求 ID 或交叉引用不一致；
- 证据场景、行号和逐字摘录不匹配；
- 总表遗漏逐场需求；
- 总表把不同类别元素错误合并；
- 总表场景集合与来源需求不一致；
- 高复杂度索引遗漏或与逐场评分矛盾。

内外景或时段未知属于警告，不会被模型或本地规则补写为事实。校验失败时仍保存 `validation.json`，但禁止生成正式导出。

本地命令还会核对当前 Prompt、Schema、契约、模型配置、阶段版本和逐场缓存键，按共享剧本顺序重排记录，并验证产物元数据指纹等于实际内容指纹。JSONL 损坏会在任何模型调用前明确阻断；上游变化或失败会把旧总表、校验和导出标为过期；总表必须等于正式构建器从当前逐场结果重建出的全部字段；现有导出必须逐字节等于当前确定性渲染，否则自动重建。

规划校验在此基础上提供三个单调递进的准备度：

- `draft_valid`：拍摄单元完整覆盖、引用双向一致、证据可定位，高危候选没有被删除；
- `catalog_ready`：在结构有效之外，不再存在疑似漏拆阻断，跨场实体和标准单位已确认，危险默认实现已明确否决；
- `shoot_ready`：在目录就绪之外，所有高危范围均按所需角色取得唯一、未过期的合格专业批准。

高一级不可能绕过低一级。规划报告中的每条问题都会列出它阻断的准备度，不用一个笼统的 `valid` 掩盖“结构可读”和“可进入专业执行流程”的差别。

## 9. CLI、修正闭环与产物

```powershell
movie-breakdown production analyze ./项目
movie-breakdown production resume ./项目
movie-breakdown production status ./项目
movie-breakdown production validate ./项目
movie-breakdown production export ./项目 --format all

# 以下命令全部复用已经验证的制片产物，不调用模型
movie-breakdown production plan ./项目
movie-breakdown production review ./项目
movie-breakdown production review ./项目 --answers ./制片专家答案.json
movie-breakdown production correct ./项目 ./制片修正集合.json `
  --answers ./制片专家答案.json --dry-run
movie-breakdown production correct ./项目 ./制片修正集合.json `
  --answers ./制片专家答案.json
movie-breakdown production review ./项目
movie-breakdown production review ./项目 --answers ./制片复核答案.json
movie-breakdown production finalize ./项目 --profile evaluation
movie-breakdown production finalize ./项目 --profile professional
```

只有 `analyze` 和 `resume` 需要 DeepSeek 密钥。`status`、`validate`、`export`、`plan`、`review`、`correct` 和 `finalize` 不读取密钥、不调用模型。

`plan` 从当前 100% 通过基础校验的逐场结果重建基础规划。`review` 生成完整强制目标和答案模板；导入答案时严格核对规划、目标集、评分标准和安全政策指纹。`correct` 要求修正操作同时绑定旧值指纹、评审目标、专家答案、理由和逐字证据；每个修正集都是相对基础规划的完整累计快照，不是叠加在上一版上的隐式增量。

必须先运行 `correct --dry-run` 完成零写入预演，再激活同一修正集。激活时先完整写入内容寻址 generation，最后原子更新 `active.json`；中途失败不会把半套产物设为正式版本。正式规划改变后必须重新执行 `review` 并填写新目标集，不能复用旧指纹答案。

`finalize` 会重算规划校验、评审目标和十项发布检查。被阻断的结论仍保存报告，但不会生成不可变发布；通过后才按全部内容指纹建立不可变归档。

### 9.1 `evaluation` 与 `professional`

两种发布等级不能混用：

- `evaluation` 要求规划达到 `catalog_ready`、强制目标 100% 评完、结论与评分完整、存在修正时回执有效，并且全部指纹匹配。AI 模拟专家可以形成明确标注的 `evaluation_ready`，但该状态不代表 `shoot_ready`，也不代表专业签署；
- `professional` 进一步要求规划达到 `shoot_ready`、评审者为真人专家，并且每个高危候选的每个指定专业角色都存在唯一、范围匹配的合格批准。只有这一级通过后状态才是 `professional_stable`；
- 任一检查失败都统一得到 `blocked`。封版只说明制片拆解底稿在所请求等级下受控，不代表预算、通告、采购订单或具体高危拍摄方案已经获批。

## 10. 当前明确不做

- 预算、货币金额和成本估计；
- 自动生成拍摄日程、通告单或场景合并排期；
- 自动推算采购、租赁、库存或备份数量；人工可以在计划层记录已经作出的决定；
- 拍摄地、棚拍/实景、供应商或演员推荐；
- 未经人工确认的同义词、简称、模糊实体归并；
- 用叙事推断替代制片原文证据；
- 让模型或 AI 模拟专家批准高危拍摄方法；
- 生产部门 Excel 模板和预算软件对接。

CSV 是通用 UTF-8 交换文件；如后续需要固定 Excel 模板，应在真实制片部门确认字段和工作流后独立实现。

## 11. 验收基线与公开边界

自动化层已经覆盖 Schema、docstring、文件行数、缓存恢复、token 累积、证据绑定、总表确定性、存储隔离、无模型命令、退出码和多格式导出。

每轮真实验收只选择一部经授权的代表性剧本，不调用或改写其他剧本。公开仓库只保留验证范围与合格判据；剧本名称、场景和实体统计、专家逐项评分、修正数量、Token 账单及账户信息均保存在不受 Git 跟踪的私有验收目录。

代表性验收必须确认全部场景成功处理且覆盖率完整，四个独立制片阶段状态一致，结构校验与多格式导出通过，并验证模型用量能够沿缓存谱系审计。自动化检查与真实在线契约测试分离，在线测试必须通过显式开关启用，避免意外调用模型或产生费用。

闭环验收中的 `production plan/review/correct/finalize` 只复用已经存在且重新校验过的制片结果，不得调用 DeepSeek，也不得改写原始逐场模型产物；闭环执行前后的可审计模型 Token 用量应保持不变。

本轮验收必须逐项留存以下证据，结果只能在实际产物生成和复验后回填，不能仅凭实现代码宣称通过：

1. 拍摄单元连续覆盖全部场景，并对疑似复合场景完成评审或结构化修正；
2. 资源类别、实体和出现项双向引用一致，跨场连续性候选均有明确确认或阻断结论；
3. 数量事实、父子语义、标准单位和人工计划相互分离，未知单位与无证据数字没有被伪装成可执行值；
4. 全量高危候选不可被删除，危险默认实现有明确否决决定，全部安全目标进入强制复核；
5. `correct --dry-run` 零写入、激活 generation 原子可重放、修正后使用新指纹完成复核；
6. AI 模拟专家最多验证 `evaluation` 路径；没有真人专家身份与逐角色合格安全批准时，`professional` 必须明确阻断，不能用模拟结果冒充专业稳定版。

真实验收发现的问题继续先固化为回归样本，再修改 Prompt、Schema 或确定性规则并分批提交。
