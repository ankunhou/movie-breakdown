## 2. 叙事产物

### 2.1 运行和中间产物

| 路径 | 内容 | 主要消费者 |
| --- | --- | --- |
| `project.json` | 项目 ID、来源、框架和固化模型配置 | `resume`、缓存判定 |
| `source/<源文件>` | 初始化时复制的项目内源 | 所有恢复和来源指纹核对 |
| `artifacts/manifest.json` | 阶段状态、版本、缓存、用量和错误 | `status`、`resume` |
| `artifacts/normalized.json` | 规范化文本和行/页映射 | 场景和证据 |
| `artifacts/scenes.json` | 场景列表、标题、原文范围和文本 | 叙事、制片共享 |
| `artifacts/scene_analysis.jsonl` | 逐场分析、尝试次数和 Token | 全局分析和恢复 |
| `artifacts/entities.json` | 归一人物、地点等实体 | 正式叙事聚合 |
| `artifacts/events.json` | 归一事件和因果 | 结构、情节线 |
| `artifacts/relationships.json` | 人物关系 | 关系发展和人物弧光 |
| `artifacts/structure.json` | 三幕、节拍、情节线、主题和母题 | 正式报告 |
| `artifacts/global_recovery.json` | 全局结构化恢复审计 | 模型失败恢复 |
| `artifacts/character_dossiers.json` | 所有归一人物的分级档案 | 检索、复核和报告 |
| `artifacts/character_biographies.jsonl` | 核心人物逐项可恢复小传 | 小传恢复 |
| `artifacts/biographies.json` | 汇总后的核心人物小传 | 正式报告 |
| `artifacts/validation.json` | 当前一致性校验 | `validate`、`export` |

### 2.2 正式导出和闭环

| 路径 | 生成命令 | 内容 |
| --- | --- | --- |
| `exports/breakdown.json` | `analyze`、`resume`、`export`、正式 `correct` | 机器可读完整叙事拆解 |
| `exports/report.md` | 同上 | 人类可读叙事报告 |
| `artifacts/semantic_quality.json` | `review` | 自动信号、目标、答案和摘要 |
| `exports/semantic-quality.md` | `review` | 风险信号、上下文和人工判断 |
| `exports/human-review-<指纹前12位>.json` | `review` | 可填写答案模板 |
| `corrections/active.json` | 正式 `correct` | 当前激活修正集合 |
| `corrections/review_answers.json` | 正式 `correct` | 修正所依据的答案 |
| `artifacts/corrected_breakdown.json` | 正式 `correct` | 修正后正式分析快照 |
| `artifacts/correction_receipt.json` | 正式 `correct` | 指纹和操作审计回执 |
| `artifacts/release_gate.json` | `finalize` | 叙事门禁机器报告 |
| `exports/release-gate.md` | `finalize` | 叙事门禁可读报告 |

`exports/breakdown.json` 的主要顶层内容：

- `screenplay`；
- `scene_analyses`；
- `entities`；
- `events`；
- `relationships`；
- `dossiers`；
- `biographies`；
- `structure`；
- `validation`；
- 可选 `correction_receipt`。

`report.md` 包含故事概览、三幕结构、全人物分级档案、核心人物小传、情节线、节拍、主题、母题、场景索引、指纹和可选修正回执。

| 路径 | 内容 |

## 4. CSV 字段和用途

### 4.1 基础双 CSV

`scenes.csv` 常用列：

- 场景 ID 和标题；
- 地点、子地点、内外景和时段；
- 演员数、群演组数和元素数；
- 复杂度分数、等级；
- 待确认数量。

`catalog.csv` 常用列：

- 目录类别和元素子类别；
- ID 和规范名称；
- 涉及场景；
- 数量下界和上界；
- 单位。

### 4.2 六张规划 CSV

| 文件 | 粒度 | 主要列和用途 |
| --- | --- | --- |
| `shooting_units.csv` | 每拍摄单元 | ID、场景、序号、标签、描述、原文行、拆分原因、地点、内外景/时段、出现项 |
| `resources.csv` | 每资源类别或实体 | 规范名、单位、身份范围、实体、解析状态、归一依据、重定向、备注 |
| `occurrences.csv` | 每资源出现 | 场景/单元、来源需求、资源类别、实体、解析状态、数量事实、前后状态、证据行 |
| `quantities.csv` | 每数量事实或计划 | 上下界、单位、角色/用途、父数量、互斥组、来源、决定和理由 |
| `safety.csv` | 每风险或决定 | 触发规则、控制项、禁止方法、角色、评审人类型/决定、范围匹配和替代政策 |
| `issues.csv` | 每校验问题 | 严重度、代码、消息、引用和被阻断准备度 |

CSV 用于审核和数据交换，不是预算、拍摄排期、通告单或采购单。

## 5. JSON 信封和 JSONL

### 5.1 普通阶段信封

多数阶段 JSON 使用：

```json
{
  "metadata": {
    "stage": "<阶段名>",
    "stage_version": "<版本>",
    "cache_key": "<内容缓存键>",
    "artifact_fingerprint": "<业务产物指纹>",
    "source_fingerprint": "<源指纹>",
    "upstream_fingerprints": [],
    "prompt_fingerprint": null,
    "schema_fingerprint": "<Schema 指纹>",
    "model": null,
    "model_parameters": {},
    "created_at": "<UTC 时间>",
    "usage": {
      "input_tokens": 0,
      "output_tokens": 0,
      "total_tokens": 0
    }
  },
  "data": {}
}
```

以下主要文件不是普通阶段信封，而是严格业务 JSON：

- `global_recovery.json`；
- `semantic_quality.json`；
- `corrected_breakdown.json`；
- `correction_receipt.json`；
- `release_gate.json`；
- `production/planning/*`；
- `production/reviews/*`；
- generation 和 immutable release 内文件；
- `production/releases/report.json`。

### 5.2 JSONL 可恢复记录

每行独立：

```json
{
  "scene_id": "<场景 ID>",
  "cache_key": "<指纹>",
  "status": "success",
  "analysis": {},
  "error": null,
  "attempts": 1,
  "usage": {
    "input_tokens": 0,
    "output_tokens": 0,
    "total_tokens": 0
  }
}
```

核心人物小传 JSONL 使用人物 ID 等对应字段，但恢复思想相同。每个单项完成后原子刷新，因此进程中断不会丢失全部进度。

## 6. 缓存与恢复

### 6.1 有效缓存条件

一个成功阶段只有同时满足下列条件才会复用：

- 当前输入缓存键一致；
- 阶段版本一致；
- 重新计算的业务产物指纹一致。

缓存身份覆盖：

- 源内容和来源位置；
- 上游产物指纹；
- 阶段版本；
- Prompt；
- Pydantic Schema；
- 模型、思考模式和推理强度；
- 关键运行参数。

因此，文件名相同不代表可以复用；内容没有变化时，改名也不必然意味着业务结果变化。

### 6.2 恢复粒度

- 逐场叙事：按场景 JSONL 恢复；
- 核心人物小传：按人物恢复；
- 逐场制片：按场景 JSONL 恢复；
- 全局叙事：保留 `global_recovery.json` 审计结构化恢复；
- 人工修正：在全量校验后原子激活，不暴露半完成正式状态。

制片逐场缓存损坏会在继续模型调用前终止并使下游过期。失败记录只有在包含可严格恢复的结构化内容时才会本地恢复，否则由 `resume` 重跑。

删除或篡改导出文件不会被阶段成功状态掩盖；导出阶段会比较确定性内容并重写。

### 6.3 过期输入

以下输入只要指纹不匹配就会拒绝：

- 专家答案；
- 修正集合；
- 修正回执；
- active generation；
- 安全批准；
- immutable release。

不要通过手工替换指纹绕过。应重新生成模板、重新评审或基于当前基础产物重建累计修正。

## 7. 关键业务语义

### 7.1 人物档案和人物小传

- 所有已归一人物都有确定性分级档案；
- 分为核心人物、重要配角、功能人物、背景/索引人物；
- 只有核心人物进行昂贵的声明级模型小传扩写；
- 小传区分 `observed`、`reported`、`inferred` 和 `unknowns`；
- 全部人物有档案，不等于所有人物都有同等深度模型小传。

### 7.2 拍摄单元

`ShootingUnit` 是场景内部可分别确认地点、时段、资源和风险的最小拍摄语义单元。它不是镜头、分镜、拍摄日、通告单或摄制组 A/B 组。

同一场的单元必须按序连续、无空档覆盖该场全部原文行。

### 7.3 跨场资源

```text
ProductionResourceClass
  └─ ProductionEntity
      └─ ResourceOccurrence
```

- 资源类别：规范名称、类别、标准单位和身份范围；
- 实体：跨场同一身份和连续性状态；
- 出现项：具体场景和拍摄单元中的一次可追溯出现；
- 同名只形成待确认候选，不会静默确认成同一实体。

### 7.4 数量与单位

- `QuantityFact` 只表达原文支持的数量、上下界、原始/标准单位、父子关系和证据；
- `PlannedQuantity` 才表达人工决定的实拍、数字扩充、采购或备份数量；
- `peak_quantity` 是跨场最大同时需求，不是跨场求和，也不是采购量；
- 系统不会把模型估算伪装成原文事实；
- 子集和总数不会自动重复相加；
- 近义单位不会未经确认自动换算。

### 7.5 高危范围

风险候选记录场景、拍摄单元、资源、触发规则、证据、控制项、禁止方法和专业角色。`scope_fingerprint` 直接绑定拍摄单元 ID、风险类别、资源出现项 ID、触发规则 ID 和证据；这些内容变化会使旧批准失效。

- AI 可以识别、要求修正或否决危险默认实现；
- AI 不能关闭专业安全风险；
- 每个风险的每个必需角色都要有当前范围的合格真人决定；
- 范围变化会使旧批准失效；
- 软件字段不等于现实资质认证。

## 8. Token、成本与隐私

`status` 和 `production status` 展示每个阶段记录的 Token 用量。CLI 不按价格换算金额；模型价格会变化，应以所配置模型当时的官方计价为准。

降低无效调用：

- 首次运行前先 `doctor`；
- 中断后用 `resume`，不要删除项目重跑；
- 用 `status` 确认失败范围；
- 格式明确时可用 `--format-detection local`，但后续叙事仍调用模型；
- 不手工篡改缓存和固化配置；
- 先修复输入格式再调用模型。

模型命令会发送完成分析所需的剧本文本。密钥不会写入项目，但敏感材料仍应遵守组织的数据处理和保密政策。

## 9. 常见问题

### 9.1 提示未设置 `DEEPSEEK_API_KEY`

1. 确认命令运行目录存在 `.env`；
2. 检查变量名拼写；
3. 运行 `doctor --no-online`；
4. 如果刚设置系统环境变量，重新打开终端；
5. 不要把无意义引号或尾部空格写入 Key。

### 9.2 `doctor --no-online` 仍失败

这是预期行为。`--no-online` 只跳过模型列表接口，不跳过 Key、本地依赖和目录写权限检查。

### 9.3 输入扩展名被拒绝

当前只支持 `.txt`、`.md` 和 `.pdf`。`.markdown`、`.docx`、`.fdx` 需要先转换。

### 9.4 PDF 读取不到正文

先确认能否在 PDF 阅读器中复制文本。扫描件先 OCR。即使有文本层，复杂双栏或错误阅读顺序也可能需要转成 TXT 后整理换行。

### 9.5 场景切分太少或太多

建议：

1. 检查场景标题是否独占一行；
2. 先使用默认 `auto`；
3. 格式明确时比较 `local`；
4. 特殊格式使用 `model`；
5. 查看 `artifacts/normalized.json` 和 `scenes.json`；
6. 不直接执行未经 CLI 本地检查的模型正则。

### 9.6 提示项目已经初始化

- 主项目使用 `resume`；
- 制片作用域使用 `production resume`；
- 独立实验应使用新目录，不覆盖原项目。

### 9.7 `status` 显示 `stale`

说明源、上游、阶段版本、Prompt、Schema 或模型参数改变。模型阶段使用 `resume`；纯本地阶段可按 `validate`、`export` 或 `plan` 重新生成。

### 9.8 修改了项目内源文件

先运行叙事 `resume` 更新规范化文本和共享场景，再运行 `production resume`。直接在旧场景上继续制片会因来源指纹不一致被拒绝。

### 9.9 `validate` 失败

读取 JSON 错误或 `issues`：

```powershell
uv run movie-breakdown validate ".\workspace\sample-project" --json
uv run movie-breakdown production validate ".\workspace\sample-project" --json
```

按 `severity`、`code`、消息和引用定位。叙事上游重建错误当前通常返回 1，制片明确无效报告返回 2。

### 9.10 专家答案过期

常见原因：

- 分析或规划已修正；
- `resume` 产生新内容；
- 叙事 `--sample-size` 改变；
- 评分标准或安全策略升级；
- 答案引用旧目标。

重新运行相应 `review`，复制新模板并重新评审。不要只替换指纹。

### 9.11 运行 `review` 后已填答案消失

不带 `--answers` 的 `review` 会生成空白正式复核状态：

- 叙事答案模板文件受保护，但 `semantic_quality.json` 会重置；
- 制片 `report.json` 和 `answers-template.json` 都会覆盖。

把填写版保存在项目外独立路径，并用 `--answers` 重新导入。

### 9.12 修正提示旧值指纹不匹配

修正针对的对象或作用域已经变化。重新读取当前正式产物和当前答案，按现版本重新生成修正。不能把旧 `expected_value_fingerprint` 手工替换后沿用旧判断。

### 9.13 `--dry-run` 通过，正式应用却失败

确认两次命令之间没有：

- 运行 `resume` 或重新规划；
- 编辑项目产物；
- 激活其他修正；
- 修改答案或修正文件；
- 其他进程同时操作同一项目。

### 9.14 `catalog_ready=false`

检查：

- 是否仍有未确认跨场实体；
- 标准单位是否合法；
- 资源引用是否完整；
- 危险默认方法是否已有明确替代决定。

### 9.15 `shoot_ready=false`

要求先达到 `catalog_ready`，然后每个当前高危范围的每个必需专业角色都要有 `qualified_professional` 决定。AI 模拟答案不能满足。

### 9.16 evaluation 被阻断

查看 `production/exports/release-evaluation.md`。常见原因：

- 强制目标没有全部评完；
- 有 `needs_correction` 或 `blocked`；
- 评分不完整；
- 修正回执与当前规划不一致；
- 危险默认方法未处理；
- 高危目标使用了 `accepted_risk`。

### 9.17 professional 被阻断

查看 `release-professional.md`。除 evaluation 数据门禁外，重点检查：

- `reviewer_kind` 是否为 `human_expert`；
- `shoot_ready` 是否为真；
- 每个风险的所有必需角色是否都有决定；
- `reviewer_kind` 是否为 `qualified_professional`；
- 决定是否绑定当前 `scope_fingerprint`；
- 条件批准是否列出控制项。

这类阻断通常表示安全门禁正常工作。

### 9.18 `--json` 没有输出 JSON

检查：

- 必填参数是否缺失；
- 枚举或 `sample-size` 是否非法；
- 配置是否在进入命令前初始化失败；
- 标准输出和标准错误是否被混合；
- `$LASTEXITCODE` 是否非 0。

自动化必须使用“退出码 + JSON 解析”双重判定。

## 10. 自动化调用建议

保守的 PowerShell 示例：

```powershell
$planRaw = uv run movie-breakdown production plan `
  ".\workspace\sample-project" `
  --json
$planExit = $LASTEXITCODE
if ($planExit -ne 0) {
  throw "制片规划草稿无效，退出码：$planExit"
}
$plan = $planRaw | ConvertFrom-Json
if (-not $plan.ok -or -not $plan.validation.draft_valid) {
  throw "制片规划输出未满足 draft_valid。"
}

$releaseRaw = uv run movie-breakdown production finalize `
  ".\workspace\sample-project" `
  --profile evaluation `
  --json
$releaseExit = $LASTEXITCODE
if ($releaseExit -ne 0) {
  throw "评测封版未通过，退出码：$releaseExit"
}
$release = $releaseRaw | ConvertFrom-Json
if (-not $release.ok -or -not $release.report.releasable) {
  throw "评测封版 JSON 未声明可发布。"
}
```

自动化原则：

- 每步检查退出码；
- 分离标准输出和标准错误；
- 失败后保留项目并使用 `resume`；
- 修正前始终 `--dry-run`；
- 修正后强制重新评审；
- 不忽略 `finalize` 的退出码；
- 不把 `evaluation_ready` 当成 `professional_stable`；
- 不把任何软件状态当成现场开拍许可。
