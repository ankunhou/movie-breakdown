# 04 专家评审与人工修正

本章说明 CLI 接受的 4 类外部 JSON：

1. 叙事专家答案；
2. 叙事修正集合；
3. 制片专家答案；
4. 制片累计修正集合。

所有 JSON 例子只用于说明结构。`<...>` 是占位符，不能直接运行。

返回[手册首页](README.md)。

## 1. 通用规则

所有外部输入继承严格 Pydantic 模型：

- 未声明字段会直接触发 `ValidationError`；
- 枚举值区分大小写；
- JSON 文件必须使用 UTF-8；
- ID、指纹、版本必须来自当前项目；
- 不要把数字写成字符串；
- 不要手工改新指纹来伪装旧答案仍然有效；
- 先复制 CLI 生成的答案模板，只编辑人工填写字段；
- 修正操作必须带当前旧值/作用域指纹和可定位证据；
- `--dry-run` 通过后再正式激活。

### 1.1 当前模板能力

CLI 会生成：

- 叙事 `HumanReviewAnswers` 模板；
- 制片 `ProductionReviewAnswers` 模板。

CLI 当前不会生成：

- 叙事 `CorrectionSet` 模板；
- 制片 `ProductionCorrectionSet` 模板；
- 修正所需的 `expected_value_fingerprint`。

修正集合属于高级、受控集成输入。指纹是项目规范 JSON 内容指纹，不能用文件字节的 `sha256sum` 替代。需要批量构造修正时，应通过项目 Pydantic 模型和 `content_fingerprint` 语义在受测试的集成脚本中生成，再交给 `--dry-run` 验证。项目内部导入路径不承诺是稳定公共 API。

### 1.2 通用 Evidence

```json
{
  "scene_id": "<证据场景 ID>",
  "source_span": {
    "line_start": 100,
    "line_end": 103,
    "page_start": null,
    "page_end": null
  },
  "excerpt": "<与对应全局行号逐字一致的原文，不超过 300 字>",
  "confidence": "high"
}
```

规则：

- `line_start` 和 `line_end` 从 1 开始，结束不能早于开始；
- PDF 可带 `page_start` 和 `page_end`，TXT/MD 通常为 `null`；
- `excerpt` 必须能由指定场景和行号逐字定位；
- `confidence` 只能是 `high`、`medium` 或 `low`。

## 2. 叙事专家答案

### 2.1 生成模板

```powershell
uv run movie-breakdown review ".\workspace\sample-project" --sample-size 16
```

模板：

```text
PROJECT/exports/human-review-<分析指纹前12位>.json
```

同指纹模板已存在时不会覆盖。正式质量状态仍会在每次 `review` 时更新。

### 2.2 顶层字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | ---: | --- |
| `schema_version` | 字符串 | 否 | 默认 `1.0`；建议保留模板原值 |
| `analysis_fingerprint` | 字符串 | 是 | 当前正式分析指纹 |
| `rubric_version` | 字符串 | 是 | 当前评分标准版本 |
| `reviewer` | 字符串 | 否 | 默认空字符串；封版要求非空、可追溯身份 |
| `responses` | 数组 | 是 | 可部分提交；未提交目标保持未评 |

### 2.3 Response 字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `target_id` | 字符串 | 必须是当前抽样中的目标 |
| `verdict` | 枚举 | 评审结论 |
| `ratings` | 数组 | 当前目标适用维度的评分 |
| `notes` | 字符串 | 判断依据或风险接受说明 |
| `proposed_correction` | 字符串或 null | 建议修正；进入 `correct` 的目标必须非空 |

`verdict`：

- `unreviewed`；
- `supported`；
- `partially_supported`；
- `unsupported`；
- `uncertain`。

Rating：

```json
{
  "dimension": "source_fidelity",
  "score": 5,
  "comment": "结论与目标附带的原文一致。"
}
```

`score` 为 1—5 或 `null`。部分评审可省略某些适用维度，但叙事封版要求每个目标的所有适用维度恰好评分一次。

可用维度：

| 值 | 判断内容 |
| --- | --- |
| `source_fidelity` | 是否忠于原文 |
| `evidence_sufficiency` | 证据是否足以支撑 |
| `causal_coherence` | 因果是否连贯 |
| `structural_plausibility` | 结构、幕、节拍或转折解释是否合理 |
| `character_arc_coherence` | 人物欲望、动机和变化是否连贯 |
| `character_portrait_coherence` | 人物小传和整体形象是否连贯 |
| `theme_plausibility` | 主题或母题是否有文本依据 |
| `uncertainty_calibration` | 是否正确区分呈现、转述、推断与未知 |

### 2.4 完整结构示例

```json
{
  "schema_version": "1.0",
  "analysis_fingerprint": "<从模板原样保留>",
  "rubric_version": "<从模板原样保留>",
  "reviewer": "<评审人姓名或受控标识>",
  "responses": [
    {
      "target_id": "<从模板原样保留>",
      "verdict": "supported",
      "ratings": [
        {
          "dimension": "source_fidelity",
          "score": 5,
          "comment": "结论与所列原文一致。"
        },
        {
          "dimension": "evidence_sufficiency",
          "score": 4,
          "comment": "证据足以支持主要判断。"
        }
      ],
      "notes": "已核对目标附带的完整场景上下文。",
      "proposed_correction": null
    }
  ]
}
```

只保留该目标模板中已有的适用 `dimension`，不要自行增加。

### 2.5 风险接受

如果使用 `partially_supported` 或 `uncertain` 并准备封版，`notes` 必须使用精确前缀：

```text
接受风险：<具体风险、影响和为什么仍接受>
```

`unsupported` 不能通过叙事封版，必须修正或重新判断。

## 3. 叙事修正集合

### 3.1 顶层字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `schema_version` | 字符串 | 当前 `1.0` |
| `source_fingerprint` | 字符串 | 当前项目来源指纹 |
| `base_analysis_fingerprint` | 字符串 | 修正前正式分析指纹 |
| `rubric_version` | 字符串 | 对应答案评分版本 |
| `review_answers_fingerprint` | 字符串 | 对应答案的规范内容指纹 |
| `reviewer` | 字符串 | 必须与答案一致且非空 |
| `corrections` | 非空数组 | 原子应用的一组修正 |

### 3.2 单条修正字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | 字符串 | 集合内唯一修正 ID |
| `review_target_id` | 字符串 | 提出修正的当前目标 |
| `field` | 枚举 | 允许修改的文本字段 |
| `object_id` | 字符串 | 目标业务对象 |
| `expected_value_fingerprint` | 字符串 | 对象当前旧字段值指纹 |
| `replacement` | 字符串或 null | 新文本；空白字符串非法 |
| `rationale` | 非空字符串 | 修正理由 |
| `evidence` | 1—12 项 | 逐字可定位证据 |

只有领域原字段本身可空时，`replacement=null` 才可能通过最终 Schema；主题和母题列表项不能用 null 删除。

### 3.3 可修正字段和对象

| `field` | `object_id` | 典型 `review_target_id` |
| --- | --- | --- |
| `scene_summary` | 场景 ID | `scene-summary:<scene-id>` |
| `event_summary` | 事件 ID | `event:<event-id>` |
| `act_summary`、`act_turning_point` | 幕数字字符串 | `act:<幕数字>` |
| `beat_summary` | 节拍 ID | `beat:<beat-id>` |
| `plot_summary` | 情节线 ID | `plot:<plot-id>` |
| `foreshadow_description` | 伏笔 ID | `foreshadow:<id>` |
| `arc_initial_state`、`arc_desire`、`arc_need`、`arc_final_state` | 人物 ID | `arc:<character-id>` |
| `relation_development` | 关系 ID | `relation:<relation-id>` |
| `biography_summary` | 人物 ID | `biography:<character-id>` |
| `biography_claim_statement`、`biography_claim_rationale` | claim ID，或人物 ID 与 claim ID 组合 | `biography-claim:<character-id>:<claim-id>` |
| `theme` | 一基序号目标 ID | `theme:<序号>` |
| `motif` | 一基序号目标 ID | `motif:<序号>` |

### 3.4 结构示例

```json
{
  "schema_version": "1.0",
  "source_fingerprint": "<当前来源指纹>",
  "base_analysis_fingerprint": "<修正前分析指纹>",
  "rubric_version": "<对应答案中的版本>",
  "review_answers_fingerprint": "<对应答案的规范内容指纹>",
  "reviewer": "<与答案完全一致>",
  "corrections": [
    {
      "id": "narrative-correction-001",
      "review_target_id": "scene-summary:<场景 ID>",
      "field": "scene_summary",
      "object_id": "<场景 ID>",
      "expected_value_fingerprint": "<当前旧摘要指纹>",
      "replacement": "<经专家确认的新摘要>",
      "rationale": "说明原摘要哪里不忠于原文以及修正依据。",
      "evidence": [
        {
          "scene_id": "<场景 ID>",
          "source_span": {
            "line_start": 100,
            "line_end": 103,
            "page_start": null,
            "page_end": null
          },
          "excerpt": "<对应行的逐字原文>",
          "confidence": "high"
        }
      ]
    }
  ]
}
```

该示例中的占位指纹不能运行。

### 3.5 导入顺序

```powershell
# 先导入提出修正的答案
uv run movie-breakdown review ".\workspace\sample-project" `
  --sample-size 16 `
  --answers ".\review\narrative-answers.json"

# 零写入预演
uv run movie-breakdown correct ".\workspace\sample-project" `
  ".\review\narrative-corrections.json" `
  --answers ".\review\narrative-answers.json" `
  --dry-run

# 预演成功后激活
uv run movie-breakdown correct ".\workspace\sample-project" `
  ".\review\narrative-corrections.json" `
  --answers ".\review\narrative-answers.json"
```

激活后必须基于新分析指纹重新生成、填写并导入答案，旧答案不能用于 `finalize`。

## 4. 制片专家答案

### 4.1 生成模板

```powershell
uv run movie-breakdown production review ".\workspace\sample-project"
```

模板：

```text
PROJECT/production/reviews/answers-template.json
```

这份模板会被后续 `production review` 覆盖。先复制到项目外的受控评审路径再填写。

### 4.2 顶层字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `schema_version` | 字符串 | 当前 `1.0` |
| `plan_fingerprint` | 字符串 | 当前正式规划指纹 |
| `target_set_fingerprint` | 字符串 | 当前完整目标集指纹 |
| `rubric_version` | 字符串 | 当前评分标准版本 |
| `safety_policy_version` | 字符串 | 当前安全策略版本 |
| `reviewer` | 字符串 | 评审者姓名或受控标识；封版要求非空 |
| `reviewer_kind` | 枚举 | `ai_simulated` 或 `human_expert` |
| `reviewer_roles` | 字符串数组 | 当前评审者承担的制片角色，不得重复 |
| `responses` | 数组 | 允许部分提交 |

`reviewer_kind` 只描述主制片评审：

- `ai_simulated`：明确标注的 AI 模拟专家，可以参与 evaluation；
- `human_expert`：真人制片专家，可满足 professional 的主评审身份要求。

它不等同于单项安全批准中的 `SafetyReviewerKind`。

### 4.3 Response 和 Rating

Response：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `target_id` | 字符串 | 当前强制目标 |
| `verdict` | 枚举 | 目标结论 |
| `ratings` | 数组 | 适用制片维度评分 |
| `notes` | 字符串 | 判断、风险或阻断说明 |
| `correction_ids` | 字符串数组 | 关联的结构化修正 ID |

`verdict`：

- `unreviewed`；
- `supported`；
- `needs_correction`；
- `accepted_risk`；
- `blocked`。

Rating：

```json
{
  "dimension": "identity_resolution",
  "score": 4,
  "comment": "跨场身份有明确连续性依据。"
}
```

可用维度：

- `source_fidelity`；
- `shooting_unit_boundary`；
- `identity_resolution`；
- `quantity_fidelity`；
- `unit_standardization`；
- `continuity`；
- `safety_scope`；
- `implementation_safety`。

`score` 为 1—5 或 `null`。封版要求所有目标的全部适用维度都填写分数。

### 4.4 结构示例

```json
{
  "schema_version": "1.0",
  "plan_fingerprint": "<从模板原样保留>",
  "target_set_fingerprint": "<从模板原样保留>",
  "rubric_version": "<从模板原样保留>",
  "safety_policy_version": "<从模板原样保留>",
  "reviewer": "<评审人姓名或受控标识>",
  "reviewer_kind": "ai_simulated",
  "reviewer_roles": ["script_breakdown_reviewer"],
  "responses": [
    {
      "target_id": "<从模板原样保留>",
      "verdict": "supported",
      "ratings": [
        {
          "dimension": "source_fidelity",
          "score": 5,
          "comment": "与场景证据一致。"
        }
      ],
      "notes": "已核对当前正式规划。",
      "correction_ids": []
    }
  ]
}
```

如果 `verdict=needs_correction`，`correction_ids` 应列出对应操作 ID；修正操作的 `review_target_ids` 必须反向包含此目标。

### 4.5 `accepted_risk` 的边界

- 普通制片目标使用时必须写具体 `notes`；
- 安全候选不能用它关闭风险；
- 危险默认实现不能用它关闭；
- `blocked`、`needs_correction`、`unreviewed` 都会阻断封版。

## 5. 制片累计修正集合

### 5.1 顶层字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `schema_version` | 字符串 | 当前 `1.0` |
| `source_fingerprint` | 字符串 | 当前项目来源指纹 |
| `base_plan_fingerprint` | 字符串 | `planning/base.json` 规范内容指纹 |
| `target_set_fingerprint` | 字符串 | 对应评审目标集指纹 |
| `review_answers_fingerprint` | 字符串 | 对应答案规范内容指纹 |
| `rubric_version` | 字符串 | 对应评分版本 |
| `safety_policy_version` | 字符串 | 对应安全策略版本 |
| `reviewer` | 字符串 | 与答案完全一致 |
| `reviewer_kind` | 枚举 | 与答案完全一致 |
| `corrections` | 非空数组 | 相对基础规划的完整累计操作 |

### 5.2 每条操作的公共字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `kind` | 枚举 | 可辨识操作类型 |
| `id` | 字符串 | 集合内唯一 |
| `review_target_ids` | 1—1000 项 | 与答案双向绑定，不得重复 |
| `expected_value_fingerprint` | 字符串 | 当前基础规划该作用域指纹 |
| `rationale` | 非空字符串 | 修正理由 |
| `evidence` | 1—24 项 | 可逐字定位的 Evidence |

### 5.3 操作专有字段

| `kind` | 额外字段 | `replacement` |
| --- | --- | --- |
| `replace_shooting_units` | `scene_id` | 该场完整 `ShootingUnit[]`，至少 1 项 |
| `replace_entity_registry` | 无 | 全局完整 `ProductionEntity[]` |
| `replace_resource_classes` | 无 | 全局完整 `ProductionResourceClass[]`，至少 1 项 |
| `replace_scene_quantities` | `scene_id` | 该场完整 `QuantityFact[]` |
| `replace_planned_quantities` | 无 | 全部 `PlannedQuantity[]` |
| `replace_safety_methods` | 无 | 全部 `SafetyMethodDecision[]` |
| `replace_safety_approvals` | 无 | 全部 `SafetyApproval[]` |

同一集合不能有两个操作修改相同 `kind + scene/global` 作用域。

### 5.4 结构示例

```json
{
  "schema_version": "1.0",
  "source_fingerprint": "<当前来源指纹>",
  "base_plan_fingerprint": "<基础规划指纹>",
  "target_set_fingerprint": "<对应答案原值>",
  "review_answers_fingerprint": "<答案规范内容指纹>",
  "rubric_version": "<对应答案原值>",
  "safety_policy_version": "<对应答案原值>",
  "reviewer": "<与答案完全一致>",
  "reviewer_kind": "human_expert",
  "corrections": [
    {
      "id": "production-correction-001",
      "kind": "replace_scene_quantities",
      "review_target_ids": ["<needs_correction 目标 ID>"],
      "expected_value_fingerprint": "<基础规划中该场数量作用域指纹>",
      "rationale": "按原文证据修正该场全部数量事实。",
      "evidence": [
        {
          "scene_id": "<场景 ID>",
          "source_span": {
            "line_start": 200,
            "line_end": 201,
            "page_start": null,
            "page_end": null
          },
          "excerpt": "<对应行的逐字原文>",
          "confidence": "high"
        }
      ],
      "scene_id": "scene-example-001",
      "replacement": [
        {
          "id": "quantity-example-001",
          "occurrence_id": "occurrence-example-001",
          "bounds": {
            "minimum": 1,
            "maximum": 1
          },
          "unit": "item",
          "raw_unit": "件",
          "raw_expression": "一件示例物品",
          "role": "total",
          "parent_quantity_id": null,
          "state": null,
          "exclusive_group": null,
          "provenance": "explicit_text",
          "derived_from_ids": [],
          "evidence": [
            {
              "scene_id": "scene-example-001",
              "source_span": {
                "line_start": 200,
                "line_end": 201,
                "page_start": null,
                "page_end": null
              },
              "excerpt": "<对应行的逐字原文>",
              "confidence": "high"
            }
          ]
        }
      ]
    }
  ]
}
```

`replacement` 中出现项 ID 的集合必须与该场当前已有数量事实所覆盖的出现项集合完全一致，并且事实 ID 唯一；有现存数量事实时，空数组不能用来删除全部事实。应复制该场当前完整数量作用域，再修改必要项。所有示例 ID、证据和占位指纹都必须替换为当前项目值。

```powershell
  --answers ".\review\production-answers.json" `

# 预演成功后激活
uv run movie-breakdown production correct ".\workspace\sample-project" `

```

## 6. 安全方法和批准

### 6.1 风险类别

确定性规则当前可重建：

- `firearm`；
- `pyrotechnics`；
- `open_flame`；
- `blade_combat`；
- `vehicle_action`；
- `animal_action`；
- `height_rigging`；
- `water_drowning`；
- `crowd_action`；
- `minor_performer`；
- `extreme_environment`；
- `prosthetic_gore`；
- `other`。

高危候选由本地规则产生，模型或普通修正无权静默删除。

### 6.2 危险方法决定

`SafetyMethodDecision` 用于明确否决危险默认实现并声明安全替代边界：

```json
{
  "id": "safety-method-decision-001",
  "scene_id": "<场景 ID>",
  "analysis_fingerprint": "<该场当前 SceneProductionAnalysis 的规范内容指纹>",
  "prohibited_method": "<明确禁止的危险默认实现>",
  "replacement_policy": "<由专业部门确认的安全替代边界>",
  "reviewer": "<评审人>",
  "reviewer_kind": "qualified_professional",
  "rationale": "<决定依据>"
}
```

`reviewer_kind` 可为：

- `ai_simulated`；
- `unverified_human`；
- `qualified_professional`。

方法决定记录“不能怎么做、替代边界是什么”，不等于单项专业批准。

### 6.3 安全批准

```json
{
  "hazard_id": "<当前风险 ID>",
  "scope_fingerprint": "<当前风险范围指纹>",
  "reviewer": "<合格专业人员>",
  "reviewer_role": "<该风险要求的专业角色>",
  "reviewer_kind": "qualified_professional",
  "decision": "approved_with_controls",
  "reason": "<专业判断依据>",
  "required_controls": ["<必须执行的控制措施>"]
}
```

`decision`：

- `requires_revision`；
- `requires_professional_review`；
- `approved_with_controls`；
- `approved`；
- `rejected`；
- `not_applicable`。

不可绕过的规则：

- 只有 `qualified_professional` 可以提交 `approved`、`approved_with_controls` 或 `not_applicable`；
- `approved_with_controls` 必须有非空 `required_controls`；
- 每个风险要求的每个专业角色都要独立覆盖；
- `scope_fingerprint` 直接绑定拍摄单元 ID、风险类别、资源出现项 ID、触发规则 ID 和证据，这些内容变化会使旧批准失效；
- 软件只验证字段声明和一致性，不认证现实资格；项目负责人必须线下核验。

## 7. 修正后的重新复核

叙事：

```powershell
uv run movie-breakdown review ".\workspace\sample-project" --sample-size 16
uv run movie-breakdown review ".\workspace\sample-project" `
  --sample-size 16 `
  --answers ".\review\narrative-final-answers.json"
uv run movie-breakdown finalize ".\workspace\sample-project"
```

制片：

```powershell
uv run movie-breakdown production review ".\workspace\sample-project"

  --answers ".\review\production-final-answers.json"
uv run movie-breakdown production finalize ".\workspace\sample-project" `
  --profile evaluation
```

不能通过替换旧答案里的指纹来跳过重新评审；旧目标可能已不再对应新内容。

## 8. 封版前自查

### 8.1 叙事

- 当前结构校验通过；
- 质量报告与当前分析指纹一致；
- 评审者非空；
- 实际目标至少 16 个且 100% 完成；
- 无 `unreviewed`、`unsupported`；
- 所有适用维度有 1—5 分；
- 部分支持或不确定结论有 `接受风险：` 说明。

### 8.2 制片 evaluation

- `catalog_ready=true`；
- 当前规划和目标集指纹匹配；
- 强制目标 100% 完成；
- 无 `unreviewed`、`needs_correction`、`blocked`；
- 所有维度有分；
- 修正回执绑定当前规划；
- 危险默认实现已明确处理；
- 高危目标全部用 `supported` 闭环；
- AI 模拟身份被明确标注。

### 8.3 制片 professional

在 evaluation 数据闭环基础上：

- `shoot_ready=true`；
- 主评审为 `human_expert`；
- 每个风险的每个必需角色恰有一条范围匹配决定；
- 决定来自 `qualified_professional`；
- 条件批准列出控制项。
