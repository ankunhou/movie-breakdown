# 03 制片命令

本章逐条说明 `production` 命名空间下 9 条命令。除 `production analyze` 和 `production resume` 外，其余命令都不读取 API Key、不调用模型。

返回[手册首页](README.md)。

## 1. `production analyze`

用途：首次创建独立制片作用域，并基于主项目共享场景执行逐场制片元素拆解。

### 1.1 语法与参数

```powershell
uv run movie-breakdown production analyze PROJECT [--json]
```

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | ---: | --- | --- |
| `PROJECT` | 路径参数 | 是 | 无 | 已完成场景切分的主项目 |
| `--json` | 布尔开关 | 否 | 关闭 | 机器可读终端输出 |

### 1.2 输入和前置条件

至少需要：

- 主项目 `project.json`；
- 项目内源文件；
- 与源指纹一致的 `artifacts/scenes.json`；
- API Key；
- 尚未存在 `production/config.json` 和 `production/manifest.json`。

不要求叙事逐场、全局结构或人物小传全部完成。制片流水线只读共享场景，并核对父项目 ID 和项目内源文件指纹。

### 1.3 拆解内容和阶段

逐场制片内容：

- 地点、子地点、内外景、时段和天气；
- 演员的画面出演、仅声音、照片/录像、替身和表演要求；
- 群演群体、数量边界和特殊技能；
- 服装、妆发、道具、陈设、车辆和动物；
- 动作特技、实拍特效、视效、特殊设备；
- 现场声音、音乐和开放兜底元素；
- 复杂度、证据因素和待确认事项。

四个隔离阶段：

1. `production_scene_analysis`；
2. `production_catalog`；
3. `production_validation`；
4. `production_export`。

### 1.4 输出、写入和退出码

成功创建：

- `production/config.json`；
- `production/manifest.json`；
- `production/artifacts/scene_elements.jsonl`；
- `production/artifacts/catalog.json`；
- `production/artifacts/validation.json`；
- `production/artifacts/breakdown.json`；
- `production/exports/breakdown.json`；
- `production/exports/report.md`；
- `production/exports/scenes.csv`；
- `production/exports/catalog.csv`。

非 JSON 模式显示进度、制片校验摘要和路径。JSON：

```json
{
  "ok": true,
  "project": "<主项目绝对路径>",
  "validation": {
    "valid": true,
    "scene_count": 0,
    "analyzed_scene_count": 0,
    "coverage": 0.0,
    "catalog_item_count": 0,
    "issues": []
  },
  "exports": {
    "json": "<breakdown.json 绝对路径>",
    "markdown": "<report.md 绝对路径>",
    "scenes_csv": "<绝对路径>",
    "catalog_csv": "<绝对路径>"
  }
}
```

字段中的数字只是结构示意。

- 完整制片拆解成功：0；
- 父项目、源、模型、Schema、作用域或流水线错误：1；
- 用法错误：2。

### 1.5 示例

```powershell
uv run movie-breakdown production analyze ".\workspace\sample-project"
```

重复初始化会返回错误并提示使用 `production resume`。

## 2. `production resume`

用途：恢复已经初始化的制片流水线，仅重试失败、未完成、缺失或过期内容。

### 2.1 语法与参数

```powershell
uv run movie-breakdown production resume PROJECT [--json]
```

| 参数 | 必填 | 说明 |
| --- | ---: | --- |
| `PROJECT` | 是 | 已有 `production/config.json` 和清单的主项目 |
| `--json` | 否 | 输出与 `production analyze` 相同的 JSON 外层 |

### 2.2 输入和恢复规则

需要：

- 当前主项目及项目内源文件；
- 未过期的共享 `artifacts/scenes.json`；
- 已初始化的制片配置和清单；
- 现有逐场缓存；
- API Key。

每场制片分析独立保存 `attempts` 和 `usage`。缓存键、阶段版本和产物内容都有效的场景不会重复调用；上游场景、Prompt、Schema、模型参数或阶段版本变化会使相关内容过期。

### 2.3 输出、写入和退出码

成功输出与 `production analyze` 相同，并刷新受影响的目录、校验和四类基础导出。

- 成功：0；
- 项目、密钥、模型、恢复或校验错误：1；
- 用法错误：2。

### 2.4 示例

```powershell
uv run movie-breakdown production status ".\workspace\sample-project"
uv run movie-breakdown production resume ".\workspace\sample-project"
```

## 3. `production status`

用途：显示四个制片阶段的状态、版本、Token 和错误。

### 3.1 语法与参数

```powershell
uv run movie-breakdown production status PROJECT [--json]
```

| 参数 | 必填 | 说明 |
| --- | ---: | --- |
| `PROJECT` | 是 | 包含 `production/manifest.json` 的主项目 |
| `--json` | 否 | 输出原始 `RunManifest` |

### 3.2 输出、副作用和退出码

终端输出与叙事 `status` 相同形式，标题为制片元素拆解阶段状态。JSON 的 `stages` 包含：

- `production_scene_analysis`；
- `production_catalog`；
- `production_validation`；
- `production_export`。

命令不读取 Key、不调用模型。通常只读取；旧清单缺少后来新增阶段时会补齐并保存。

- 成功：0；
- 制片清单缺失或损坏：1；
- 用法错误：2。

### 3.3 示例

```powershell
uv run movie-breakdown production status ".\workspace\sample-project" --json
```

## 4. `production validate`

用途：纯本地重建基础制片目录并重新校验，不生成正式导出。

### 4.1 语法与参数

```powershell
uv run movie-breakdown production validate PROJECT [--json]
```

| 参数 | 必填 | 说明 |
| --- | ---: | --- |
| `PROJECT` | 是 | 已初始化制片作用域的主项目 |
| `--json` | 否 | 输出原始 `ProductionValidationReport` |

### 4.2 输入和校验

读取并核对：

- 制片配置和清单；
- 当前父项目、项目内源文件和共享场景；
- `production/artifacts/scene_elements.jsonl`；
- 场景覆盖、证据和目录引用。

主要检查每个共享场景都有可信记录、元素 ID 引用有效、证据行属于相应场景、数量边界和制片复杂度有效。

### 4.3 输出、写入和退出码

命令会重建并写：

- `production/artifacts/catalog.json`；
- `production/artifacts/validation.json`；
- `production/manifest.json`。

旧导出阶段可能被标为过期；命令本身不写正式导出。

终端显示通过/未通过、场景覆盖和目录项目数。JSON：

```json
{
  "valid": true,
  "scene_count": 0,
  "analyzed_scene_count": 0,
  "coverage": 0.0,
  "catalog_item_count": 0,
  "issues": []
}
```

- 有效：0；
- 得到 `valid=false` 报告：先输出和保存报告，再返回 2；
- 文件、Schema、父项目或作用域错误：1；
- 用法错误：2。

### 4.4 示例

```powershell
uv run movie-breakdown production validate ".\workspace\sample-project"
```

## 5. `production export`

用途：从可信、完整的逐场制片记录重新生成基础制片报告。

### 5.1 语法与参数

```powershell
uv run movie-breakdown production export PROJECT `
  [--format markdown|json|csv|all] `
  [--json]
```

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | ---: | --- | --- |
| `PROJECT` | 路径参数 | 是 | 无 | 已初始化制片作用域的主项目 |
| `--format` | 枚举 | 否 | `markdown` | 基础制片导出格式 |
| `--json` | 布尔开关 | 否 | 关闭 | 输出本次路径映射 |

### 5.2 格式和文件

命令先重新校验，并刷新 `production/artifacts/breakdown.json`。

| `--format` | 本次正式导出 |
| --- | --- |
| `markdown` | `production/exports/report.md` |
| `json` | `production/exports/breakdown.json` |
| `csv` | `production/exports/scenes.csv` 和 `catalog.csv` |
| `all` | 同时写上述四个文件 |

这里的 `csv` 是基础制片双 CSV；`production plan` 会另写六张规划 CSV。

JSON：

```json
{
  "ok": true,
  "exports": {
    "json": "<绝对路径>",
    "markdown": "<绝对路径>",
    "scenes_csv": "<绝对路径>",
    "catalog_csv": "<绝对路径>"
  }
}
```

`exports` 只包含本次选择的格式。

### 5.3 退出码与示例

- 成功：0；
- 逐场记录不完整、校验或写入错误：1；
- 用法错误：2。

```powershell
uv run movie-breakdown production export ".\workspace\sample-project" --format all
```

## 6. `production plan`

用途：从有效基础制片拆解纯本地派生拍摄单元、跨场实体、数量语义、安全范围和三级准备度。

### 6.1 语法与参数

```powershell
uv run movie-breakdown production plan PROJECT [--json]
```

| 参数 | 必填 | 说明 |
| --- | ---: | --- |
| `PROJECT` | 是 | 已完成基础制片拆解的主项目 |
| `--json` | 否 | 输出规划指纹、准备度和路径 |

### 6.2 输入和派生对象

要求 `scene_elements.jsonl` 对共享场景达到 100% 可信覆盖；命令直接使用严格基础产物，不依赖旧的 `production/exports/*`。

派生：

| 对象 | 语义 |
| --- | --- |
| `ShootingUnit` | 场景内连续、无空档覆盖原文的最小拍摄语义单元 |
| `ProductionResourceClass` | 资源规范名、类别、标准单位和身份范围 |
| `ProductionEntity` | 跨场保持同一身份或连续性状态的实体 |
| `ResourceOccurrence` | 资源在场景和拍摄单元内的一次出现 |
| `QuantityFact` | 剧本原文能够支持的数量事实 |
| `PlannedQuantity` | 人工决定的实拍、扩充、采购或备份数量 |
| `SafetyHazard` | 本地规则重建的不可静默删除风险范围 |
| `SafetyMethodDecision` | 明确否决危险默认实现的决定 |
| `SafetyApproval` | 专业角色对不可变风险范围的决定 |

### 6.3 三级准备度

| 字段 | 通过条件摘要 |
| --- | --- |
| `draft_valid` | 场景覆盖、单元连续性、引用、数量树、证据和 Schema 有效 |
| `catalog_ready` | 草稿有效，实体和单位已确认，危险默认实现已处理 |
| `shoot_ready` | 目录就绪，且每个风险的每个必需角色都有当前范围的合格专业批准 |

`shoot_ready` 是数据门禁字段，不自动等于现场允许开拍。

### 6.4 输出、写入和退出码

每次都会覆盖保存：

- `production/planning/base.json`；
- `production/planning/validation.json`。

同时写八个导出：

1. `production/exports/planning.json`；
2. `production/exports/planning-report.md`；
3. `shooting_units.csv`；
4. `resources.csv`；
5. `occurrences.csv`；
6. `quantities.csv`；
7. `safety.csv`；
8. `issues.csv`。

JSON：

```json
{
  "ok": true,
  "plan_fingerprint": "<当前规划指纹>",
  "validation": {
    "draft_valid": true,
    "catalog_ready": false,
    "shoot_ready": false
  },
  "exports": {
    "planning.json": "<绝对路径>",
    "planning-report.md": "<绝对路径>",
    "shooting_units.csv": "<绝对路径>",
    "resources.csv": "<绝对路径>",
    "occurrences.csv": "<绝对路径>",
    "quantities.csv": "<绝对路径>",
    "safety.csv": "<绝对路径>",
    "issues.csv": "<绝对路径>"
  }
}
```

`ok` 只等于 `draft_valid`：

- `draft_valid=true`：退出 0，即使更高两级为假；
- `draft_valid=false`：保存诊断产物、输出报告后返回 2；
- 基础产物、Schema 或作用域错误：1；
- 用法错误：2。

重建基础规划后，如果旧 active generation 绑定旧基础指纹，程序会拒绝复用；需要基于新规划重新评审和生成累计修正集。

### 6.5 示例

```powershell
uv run movie-breakdown production plan ".\workspace\sample-project"
```

## 7. `production review`

用途：为当前正式规划生成全量强制复核目标和答案模板，或导入专家答案。

### 7.1 语法与参数

```powershell
uv run movie-breakdown production review PROJECT `
  [--answers PATH] `
  [--json]
```

| 参数 | 必填 | 说明 |
| --- | ---: | --- |
| `PROJECT` | 是 | 已执行 `production plan` 的项目 |
| `--answers PATH` | 否 | UTF-8 严格专家答案 JSON |
| `--json` | 否 | 输出完整报告、答案和路径 |

### 7.2 目标和答案

不存在 active generation 时，命令评审 `planning/base.json`；存在时会严格校验其基础指纹，校验有效才评审修正后的正式规划，过期或损坏则报错，不会静默回退。

强制目标覆盖：

- 每个场景的拍摄单元边界；
- 状态不是 `confirmed` 的实体；
- 按场聚合、来源为 `unknown` 或标准单位为 `unknown` 的数量事实；
- 高危候选；
- 危险默认实现。

省略 `--answers` 时，生成全量空白答案：

- `reviewer_kind=ai_simulated`；
- 所有 `responses.verdict=unreviewed`；
- 写并覆盖 `production/reviews/report.json`；
- 写并覆盖 `production/reviews/answers-template.json`。

因此，在已经保存完整答案后，不要无意运行一次不带 `--answers` 的命令。

导入答案时严格匹配：

- 当前 `plan_fingerprint`；
- 当前 `target_set_fingerprint`；
- 评分标准版本，当前 `1.0`；
- 安全策略版本，当前 `1.0`；
- 目标和适用评分维度。

允许部分 responses；不允许未知或重复目标、重复或不适用维度、重复评审角色。

### 7.3 结论和维度

结论：

| 值 | 含义 |
| --- | --- |
| `unreviewed` | 未评审 |
| `supported` | 当前规划可支持 |
| `needs_correction` | 必须关联结构化修正 |
| `accepted_risk` | 明确接受已披露风险，但不能绕过关键安全门禁 |
| `blocked` | 阻断封版 |

维度：

- `source_fidelity`；
- `shooting_unit_boundary`；
- `identity_resolution`；
- `quantity_fidelity`；
- `unit_standardization`；
- `continuity`；
- `safety_scope`；
- `implementation_safety`。

### 7.4 输出和退出码

终端显示已评数、目标数和是否完成。JSON：

```json
{
  "ok": true,
  "report": {
    "plan_fingerprint": "<当前规划指纹>",
    "target_set_fingerprint": "<目标集指纹>",
    "reviewed_count": 0,
    "target_count": 0,
    "coverage": 0.0,
    "complete": false,
    "blocked_target_ids": []
  },
  "answers": {
    "responses": []
  },
  "paths": {
    "report": "<production/reviews/report.json 绝对路径>",
    "answers": "<production/reviews/answers-template.json 绝对路径>"
  }
}
```

`answers` 和 `report` 实际包含完整严格对象，上例只展示常用字段。

- 生成或导入成功，即使 incomplete 或存在 blocked：0；
- 指纹、答案 Schema 或规划错误：1；
- 用法错误：2。

### 7.5 示例

```powershell
# 生成空白答案
uv run movie-breakdown production review ".\workspace\sample-project"

# 导入已填写答案
uv run movie-breakdown production review ".\workspace\sample-project" `
  --answers ".\review\production-answers.json"
```

## 8. `production correct`

用途：把制片专家答案与完整累计修正集绑定，预演或原子激活新 generation。

### 8.1 语法与参数

```powershell
uv run movie-breakdown production correct PROJECT CORRECTIONS `
  --answers ANSWERS `
  [--dry-run] `
  [--json]
```

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | ---: | --- | --- |
| `PROJECT` | 路径参数 | 是 | 无 | 已生成基础规划和复核的项目 |
| `CORRECTIONS` | 路径参数 | 是 | 无 | UTF-8 严格累计修正集 |
| `--answers ANSWERS` | 路径选项 | 是 | 无 | 产生这些修正的答案 |
| `--dry-run` | 布尔开关 | 否 | 关闭 | 完整预检且零写入 |
| `--json` | 布尔开关 | 否 | 关闭 | 输出校验、回执和 generation |

### 8.2 完整累计语义

修正永远以 `production/planning/base.json` 为基线应用，不是在当前 official plan 上隐式叠加。即使已有 active generation，新修正文件也必须描述相对基础规划的完整累计结果。

支持的原子操作：

| `kind` | `replacement` 必须覆盖的完整作用域 |
| --- | --- |
| `replace_shooting_units` | 指定场景的全部拍摄单元 |
| `replace_entity_registry` | 全局完整实体注册表 |
| `replace_resource_classes` | 全局完整资源类别表，稳定 ID 集合不可随意改变 |
| `replace_scene_quantities` | 指定场景的全部剧本数量事实 |
| `replace_planned_quantities` | 全部人工计划数量 |
| `replace_safety_methods` | 全部危险方法决定 |
| `replace_safety_approvals` | 全部安全批准 |

局部只提交一个新增项而省略同一作用域原有项，会被解释为删除其他项或直接校验失败。

### 8.3 绑定和校验

修正集必须匹配：

- 来源和基础规划指纹；
- 目标集和答案内容指纹；
- 评分标准和安全策略版本；
- 评审者及类型；
- 每个被替换作用域的旧值指纹。

每条操作与 `needs_correction` response 的 `correction_ids` 双向绑定，证据必须逐字定位。修正后检查：

- 场景全部原文行连续、无空档覆盖；
- 单元顺序、地点、资源和风险引用；
- 资源类别、实体和出现项；
- 数量上下界、父子关系、来源和单位；
- 高危重建、危险方法和批准权限；
- 至少达到 `draft_valid`。

### 8.4 预演、generation 和输出

`--dry-run`：

- 完全在内存校验；
- 不写闭环产物；
- `generation_id=null`；
- `exports={}`。

正式激活：

- 写内容寻址的 `production/corrections/generations/<generation-id>/`；
- generation 内保存修正集、答案、回执、正式规划、校验和 manifest；
- 最后原子更新 `production/corrections/active.json`；
- 刷新 `production plan` 的八个导出。

JSON：

```json
{
  "ok": true,
  "dry_run": false,
  "validation": {
    "draft_valid": true,
    "catalog_ready": false,
    "shoot_ready": false
  },
  "receipt": {
    "applied_correction_ids": ["<修正 ID>"],
    "applied_count": 1
  },
  "generation_id": "<generation ID>",
  "exports": {
    "planning.json": "<绝对路径>",
    "planning-report.md": "<绝对路径>",
    "shooting_units.csv": "<绝对路径>",
    "resources.csv": "<绝对路径>",
    "occurrences.csv": "<绝对路径>",
    "quantities.csv": "<绝对路径>",
    "safety.csv": "<绝对路径>",
    "issues.csv": "<绝对路径>"
  }
}
```

只要修正后 `draft_valid=true`，命令可以成功，即使 `catalog_ready` 或 `shoot_ready` 仍为假。

### 8.5 退出码与示例

- 预演或激活成功：0；
- Schema、绑定、指纹、证据或最终规划错误：1；
- 用法错误：2。

```powershell
# 零写入预演
uv run movie-breakdown production correct ".\workspace\sample-project" `
  ".\review\production-corrections.json" `
  --answers ".\review\production-answers.json" `
  --dry-run

# 预演成功后激活
uv run movie-breakdown production correct ".\workspace\sample-project" `
  ".\review\production-corrections.json" `
  --answers ".\review\production-answers.json"
```

激活后规划指纹改变，必须重新运行 `production review` 并导入新答案。

## 9. `production finalize`

用途：按必填 Profile 执行评测版或专业稳定版门禁，并始终保存门禁报告。

### 9.1 语法与参数

```powershell
uv run movie-breakdown production finalize PROJECT `
  --profile evaluation|professional `
  [--json]
```

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | ---: | --- |
| `PROJECT` | 路径参数 | 是 | 已保存当前正式规划复核的项目 |
| `--profile` | 枚举选项 | 是 | `evaluation` 或 `professional` |
| `--json` | 布尔开关 | 否 | 输出完整门禁报告 |

修正后如果保存的 review 仍绑定旧规划，命令会按过期输入错误返回 1；必须先重新评审。

### 9.2 两种 Profile

| Profile | 额外身份与准备度 | 通过状态 | 明确不代表 |
| --- | --- | --- | --- |
| `evaluation` | `catalog_ready`；主评审可为明确标注的 AI 模拟或真人 | `evaluation_ready` | 开拍或专业安全批准 |
| `professional` | `shoot_ready`；主评审必须是 `human_expert`；逐风险专业批准齐全 | `professional_stable` | 政府、保险、预算、场地或现场许可 |

### 9.3 门禁

两种 Profile 都检查：

1. 规划校验和所需准备度；
2. 规划指纹；
3. 目标集指纹；
4. 全部强制目标 100% 完成；
5. 无 `unreviewed`、`needs_correction` 或 `blocked`；
6. 所有适用维度均有 1—5 分；
7. 修正 ID 有绑定当前规划的回执；
8. 评审者身份非空；
9. 危险默认实现已明确否决；
10. 高危目标全部闭环。

补充约束：

- 非安全目标的 `accepted_risk` 必须有说明；
- 安全候选和危险默认值不能用 `accepted_risk` 关闭；
- `professional` 的每个“风险 × 必需角色”都要有范围匹配的 `qualified_professional` 决定；
- 可关闭风险的决定仅为 `approved`、`approved_with_controls` 或 `not_applicable`；
- `approved_with_controls` 必须列出控制项。

### 9.4 输出、不可变发布和退出码

无论通过或阻断，都写：

- `production/releases/report.json`；
- `production/exports/release-<profile>.json`；
- `production/exports/release-<profile>.md`。

只有通过才：

- 生成 64 个十六进制字符的内容寻址 `release_id`；
- 写 `production/releases/immutable/<release-id>/` 不可变快照。

JSON：

```json
{
  "ok": false,
  "report": {
    "profile": "professional",
    "state": "blocked",
    "releasable": false,
    "checks": []
  },
  "release_id": null,
  "exports": {
    "release-professional.json": "<绝对路径>",
    "release-professional.md": "<绝对路径>"
  }
}
```

- 门禁通过：0；
- 逻辑门禁阻断：保存报告后返回 2；
- 当前复核缺失/过期、文件或 Schema 错误：1；
- `--profile` 缺失或非法：2。

### 9.5 示例

```powershell
# 评测封版
uv run movie-breakdown production finalize ".\workspace\sample-project" `
  --profile evaluation

# 专业稳定版门禁
uv run movie-breakdown production finalize ".\workspace\sample-project" `
  --profile professional `
  --json
```
